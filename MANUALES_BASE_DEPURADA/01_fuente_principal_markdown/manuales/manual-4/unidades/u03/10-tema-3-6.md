# 3.6 Versionado y control de cambios

- Hoy es martes y tu dashboard funciona.
- El miércoles iteras 8 veces y termina mejor.
- El jueves iteras 5 veces y termina **peor**.
- ¿Cómo recuperas el bueno del miércoles?
- Sin versionado, es un dolor de cabeza. Con versionado, son dos clicks.

::concepto{titulo="Versionado de Artifacts"}
El **versionado de Artifacts** combina dos capas complementarias:

1. **Versionado interno del proveedor**
   - Claude o Gemini guardan automáticamente cada iteración.
   - Navegas entre v1, v2, v3 dentro del chat.
2. **Versionado externo en repositorio (Git)**
   - Descargas el código y lo guardas con etiquetas (semver).
   - Agregas un CHANGELOG y commits descriptivos.

- La capa 1 es cómoda pero frágil (depende del proveedor).
- La capa 2 es duradera.
- Equipos serios usan **ambas en tándem**.
::/concepto

::interioriza{titulo="Baja Curva: El check-point del videojuego"}
- Imagina que juegas un videojuego difícil (el dashboard de 6 meses).
- La Capa 1 es el "auto-guardado" temporal del nivel. Si apagas la consola, puede perderse.
- La Capa 2 es guardar la partida manualmente en tu Memory Card.
- Si un jefe (un bug) te derrota, cargas la partida guardada y no pierdes meses de progreso.
::/interioriza

## Capa 1: versiones internas del proveedor

- En Claude ves un selector v1, v2, v3 con flechas ←/→.
- Cada edición del modelo crea una versión.

**Operaciones útiles:**
- **Restaurar versión previa:** Usas flechas o click. El siguiente edit parte de ahí.
- **Comparar visualmente:** Abres pestañas separadas de distintas versiones.
- **Bifurcar:** Restauras una versión y creas una nueva línea (v3 → v4').

**Riesgos latentes:**
- El historial puede estar limitado por el proveedor.
- Al cerrar el chat, a veces se pierden versiones intermedias.
- No hay un *diff* de código exacto, solo comparación visual.

## Capa 2: versionado externo (Git)

- Descargas y haces commit cuando tienes una versión "estable".
- Ej: lista para mostrar en una reunión.

**Estructura sugerida:**
```bash
mi-asistente/
├── tools/
│   └── dashboard/
│       ├── v0.1.0/
│       │   ├── App.jsx
│       │   └── snapshot.png
│       ├── v1.0.0/
│       └── CHANGELOG.md
```

**Convenciones clave:**
- **Semver:** Mayor (breaking), menor (features), patch (fixes).
- **CHANGELOG humano:** "v0.2.0: agrega filtro. Breaking: fecha es array."
- **Snapshot (captura visual):** Para recordar qué era cada versión.
- **Commits descriptivos:** `feat: tab de fuentes`, `fix: cálculo tasa`.

## Política mínima del equipo (1 página)

- Define cuándo "publicamos" una versión (stakeholders, deploy).
- Qué hacemos al publicar:
  1. Descargar código.
  2. Crear carpeta versionada.
  3. Guardar `snapshot.png`.
  4. Actualizar `CHANGELOG.md`.
  5. Commit en Git con tag.
- Restauración rápida desde el proveedor o Git (`git checkout v0.1.0`).

## Anti-patrones costosos

- **Solo confiar en la capa interna:** Si el proveedor cae, pierdes tu trabajo.
- **Descargar sin etiquetar:** Archivos `App-final-FINAL.jsx` inmanejables.
- **CHANGELOG vacío:** En 4 versiones nadie sabrá qué cambió.
- **Falta de snapshot visual:** Difícil recordar cambios visuales.
- **Versionar cada iteración:** Genera ruido; versiona solo "estables".

## Caso real: la regresión del jueves

- **Martes:** v1.0 publicada. Funciona.
- **Miércoles:** v1.1 agrega tab.
- **Jueves:** v1.2 agrega dark mode y **rompe responsividad**.
- **Viernes:** Reportan el bug en mobile.

- **Sin Git:** Horas buscando qué rompió el diseño.
- **Con Git:** `git checkout v1.1` en 30 segundos.
- **Resultado:** Problema resuelto, equipo salva horas y mantiene confianza.

## Tres reglas de oro

1. **Versiona al publicar.** Si alguien lo verá, merece versión.
2. **Snapshot siempre.** Una imagen vale más que mil líneas de código.
3. **CHANGELOG en humano.** Escribe para tu "yo" del futuro.
