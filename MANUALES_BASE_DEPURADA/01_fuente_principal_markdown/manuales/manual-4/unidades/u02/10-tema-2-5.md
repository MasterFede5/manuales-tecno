# 2.5 Spec-driven prompting

Tu spec dice "el Asistente clasifica en 5 categorías".
Tres semanas después, alguien añade una sexta categoría.
Si olvida actualizar el prompt, producción miente sobre lo escrito.
Spec-driven prompting elimina esta grieta.

::concepto{titulo="Spec-driven prompting"}
**Spec-driven prompting** es derivar el prompt directamente desde la especificación.
Automatiza la traducción spec → prompt para que siempre estén sincronizados.

Tres formas concretas:
- **Spec como prompt:** el prompt es la spec, en lenguaje comprensible.
- **Generador spec→prompt:** un meta-prompt produce el prompt operativo.
- **Spec ejecutable:** un script genera prompt, tests y dashboard.

El objetivo es tener una sola fuente de verdad.
Si la spec cambia, el prompt cambia automáticamente.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Imagina una receta en una libreta y los ingredientes listos.
Si actualizas la receta pero no avisas al cocinero, el plato saldrá mal.
Si el cocinero **lee la receta cada vez**, no hay desfase.
Spec-driven prompting hace que tu prompt "lea la spec" cada vez.
::/interioriza

## Forma 1 — La spec ES el prompt

Es el caso más simple y potente.
La spec se escribe como un prompt XML estructurado en el repo.
La "documentación" y el "código" son exactamente el mismo archivo.

```xml
<!-- specs/clasificador-solicitudes.xml -->
<role>Eres el clasificador del Asistente Institucional Albatros.</role>

<categories>
- inscripciones: admisiones, fechas, requisitos.
- calificaciones: notas, boletas, revisiones.
- becas: descuentos, apoyos.
- queja: malestar manifiesto que requiere de-escalación.
- otro: fuera de las 4 anteriores.
</categories>

<rules>
- Prioriza la categoría más específica.
- Si hay tono molesto, marcalo.
- Si la confianza es baja, marca requiere_humano: true.
</rules>
```

Lo legible para un humano es ejecutable por la máquina.
Al actualizar la spec, actualizas el prompt directamente.

## Forma 2 — Generador spec→prompt

Útil cuando la spec es compleja y extensa.
Un script consume la spec y genera los artefactos.

```yaml
# specs/clasificador.yaml
modulo: clasificador-solicitudes
proposito: Clasificar dudas en 5 categorías
entrada: texto libre del padre/estudiante
salida: JSON con categoria, confianza, requiere_humano
categorias: [inscripciones, calificaciones, becas, queja, otro]
```

Un script `generate.py` lee este YAML.
Produce el prompt XML, los tests y el dashboard.
Toda la cadena se regenera al cambiar el YAML.

## Forma 3 — Spec ejecutable

Es la forma más madura de spec-driven prompting.
La spec es un Markdown con frontmatter ejecutable.
Un comando `npm run build` produce todos los artefactos.

```markdown
---
modulo: redactor-comunicado
version: 1.0
---

# Redactor de Comunicados

## Reglas
1. Tono "usted", sin emojis.
2. Máximo 200 palabras.

## Ejemplos golden
::ejemplo
input: motivo=mantenimiento agua
output: Estimadas familias: les comunicamos...
::/ejemplo
```

## El impacto en el equipo

Antes, el equipo tenía tres copias desincronizadas.
Producto en Notion, ingeniería en otro repo, y evaluación en otra hoja.
Con spec-driven, hay una spec y un comando.
Un cambio de dirección se refleja en todo el sistema en 5 minutos.

## Anti-patrones del spec-driven

- **Generador con magia oscura:** si nadie entiende el script, hay opacidad.
- **Spec en lenguaje no procesable:** evita descripciones ambiguas sin reglas.
- **Spec sin versión:** sin semver no puedes rastrear los prompts.
- **Regenerar todo siempre:** consume tokens innecesarios, usa diffs.

## Recomendación para tu Asistente

- **Inicia con la Forma 1** para módulos simples (clasificadores).
- **Migra a la Forma 2** cuando tengas múltiples módulos.
- **Reserva la Forma 3** para equipos grandes y onboarding rápido.

## Caso real: Cambio de spec

Director pide agregar la categoría 'salud' con escalación.
- 9:00: Agregas la categoría a la spec (5 min).
- 9:05: `npm run build` regenera prompt y tests.
- 9:10: `npm test` valida todo automáticamente.
- 10:00: Deploy a producción sin sorpresas.
