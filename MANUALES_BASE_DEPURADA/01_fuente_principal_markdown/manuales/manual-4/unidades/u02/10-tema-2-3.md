# 2.3 Definition of Done para tareas con IA

Tu tutor te dice "ya quedó". ¿Quedó? 
- ¿Pasa el smoke test? Sí.
- ¿Tiene golden set o rollback? No.
- **Conclusión:** No quedó. En IA, las reglas son distintas.

::concepto{titulo="Definition of Done (DoD) para tareas con IA"}
**Definition of Done (DoD)** exige condiciones estrictas antes de terminar una entrega.
En IA, las pruebas tradicionales no bastan. Se agregan obligaciones críticas:

- **Calidad probabilística:** El output varía; exige métricas (golden, percentiles).
- **Costo y latencia:** Variables operativas externas bajo monitoreo.
- **Rollback:** Comando único para volver a versión previa.
- **Auditoría:** Logs rastreables a 30 días.
- **Política:** Reglas claras sobre datos procesados.

Si omites el DoD, la IA fallará en casos límite imprevistos.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Piensa en el servicio de un paracaídas. 
El técnico no sólo mira si la tela "se ve bien".
Hace tres revisiones estrictas: visual, pliegues y cuerdas. 
Si falta una, no saltas. Con IA, una sola prueba es lanzarse sin paracaídas.
::/interioriza

## DoD canónico (9 puntos)

```markdown
□ TÉCNICO
  1. Prompt versionado (CHANGELOG).
  2. Schema de output validado.
  3. Smoke test pasa (3 inputs).

□ EVALUATIVO
  4. Golden set (≥15 ejemplos) supera baseline.
  5. Métricas: precisión, latencia, costo.
  6. Evaluación cruzada externa.

□ OPERATIVO
  7. Rollback documentado/probado.
  8. Política de privacidad en /docs.
  9. Dashboard de monitoreo listo.
```

**Regla de oro:** Cada punto requiere evidencia objetiva (URL, captura). Sin evidencia, no se marca.

## DoD vs Criterios de aceptación

- **Criterios de aceptación:** Lo que exige *esta* entrega específica.
- **DoD:** El estándar general fijo para *todas* las entregas.
- **Requisito:** Cumplir AMBOS para entregar un módulo.

## Plan de rollback: El salvavidas

Tu prompt v2.0 falla en producción. ¿Qué haces?
- **Sin rollback:** Pánico y pérdida de horas de equipo.
- **Con rollback:** Ejecutas `./deploy <modulo> v1.5` en ≤2 minutos.

## Política de uso (Mínima)

Añade en `/docs/política-uso.md` un párrafo por módulo:

```
[redactor-comunicado]
- Datos procesados: motivo, fecha, alcance.
- Datos NO procesados: nombres, expedientes.
- Retención/LLM: 30 días, Anthropic Claude (sin entreno).
- Revisión: 2026-04-28 por [Nombre].
```

Esto evita conflictos legales y agiliza auditorías.

## Anti-patrones del DoD

- **A ojo:** Sin evidencias tangibles.
- **A medida:** Diferente por cada módulo.
- **Auto-revisado:** Conflicto de interés obvio.
- **Post-hoc:** Ajustado para que apruebe lo ya hecho.

## Cómo implementarlo

1. **Presenta** el DoD al equipo (1 hora).
2. **Aplica** retroactivamente a 3 entregas (2 horas).
3. **Acuerda** exigirlo en la próxima entrega.
4. **Refina** el estándar tras 5 ciclos.
