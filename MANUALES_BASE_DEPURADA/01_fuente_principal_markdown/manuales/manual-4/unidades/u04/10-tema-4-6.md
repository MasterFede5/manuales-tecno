# 4.6 RAG vs fine-tuning: cuándo conviene

Tu Asistente sigue equivocándose con el lenguaje de tu institución aunque tenga RAG.
Tu desarrollador propone fine-tuning.
**Antes de gastar $500, decide si realmente lo necesitas.**

::concepto{titulo="Fine-tuning"}
**Fine-tuning** es **continuar el entrenamiento** de un modelo pre-entrenado con tus datos propios.
Ajusta sus pesos para internalizar estilo, formato o conocimiento.
Resulta en un modelo igual de potente, pero sesgado hacia tu dominio.

Tres tipos:
- **Full fine-tuning** — todos los parámetros se actualizan. Caro y raro.
- **LoRA / QLoRA** — solo capas adicionales pequeñas. Estándar moderno.
- **Adapter / Prompt tuning** — capa muy ligera. Económico.

Fine-tuning **no agrega "nuevos hechos"** de manera fiable.
Solo agrega **estilo, formato y comportamiento**.
::/concepto

::interioriza{titulo="El cocinero y la receta"}
Cuando un nuevo cocinero entra a tu restaurante, no le rehaces el cerebro.
Le **muestras los menús** (RAG) o le **das un curso de tu estilo** (fine-tuning).
Mostrar menús es barato y fácil de cambiar; el curso es caro pero funciona de memoria.
::/interioriza

## Comparativa nuclear

| Dimensión | RAG | Fine-tuning |
|---|---|---|
| **Qué resuelve** | Conocimiento factual específico | Estilo, formato, comportamiento |
| **Costo inicial** | Bajo (horas de setup) | Alto (USD 50-2000+) |
| **Actualización** | Re-indexar (minutos) | Re-entrenar (horas/días) |
| **Datos requeridos**| Documentos | Pares prompt-respuesta etiquetados |
| **Auditoría** | Alta (citas) | Baja (pesos opacos) |

## Cuándo USAR RAG (90 % de los casos)

- Conocimiento factual de tus documentos.
- Cambia con frecuencia (semanal, mensual).
- Necesitas citas verificables.
- Presupuesto limitado.

## Cuándo USAR fine-tuning

- **Estilo de escritura específico** que no se logra con prompts.
- **Tareas repetitivas** con formato fijo (ej. JSON).
- **Dominio con jerga densa** (medicina, derecho).
- **Latencia crítica** (RAG es muy lento).

## Cuándo USAR ambos

- Modelo fine-tuned para entender la **jerga**.
- RAG para recuperar los **hechos**.
- Ejemplo: IA con jerga clínica (fine-tuning) que lee historias clínicas (RAG).

## El error de "no sabe mi institución"

Si el problema es que el modelo no conoce tu reglamento, fine-tuning es una mala idea:
- Cuesta cientos de dólares y tarda días.
- Si el reglamento cambia, vuelves a pagar.
- No puede citar páginas exactas.

**RAG es objetivamente mejor para inyectar conocimiento.**

## Costos reales y herramientas (2025)

- **OpenAI**: ~$25/M tokens de entrenamiento (GPT-4o/mini).
- **Anthropic**: Claude fine-tuning en beta enterprise.
- **Local**: Ollama + LoRA es gratis (GPU local), tarda horas.
- **No-code**: OpenAI dashboard, Together AI, Modal.

## Anti-patrones

- **Datos cambiantes:** Cada cambio exige re-entrenar.
- **Dataset pequeño:** Menos de 100 ejemplos causa pobre resultado.
- **Sin baseline:** No medir el modelo base antes de fine-tunear.
- **Moda:** Hacerlo "porque suena bien" sin justificar costo.

## Recomendación final

1. **Fase inicial:** Empieza siempre con RAG.
2. **Fase avanzada:** Prueba mejorar prompts o usar pocos ejemplos (few-shot).
3. **Último recurso:** Usa fine-tuning solo si falla el tono o el formato.
