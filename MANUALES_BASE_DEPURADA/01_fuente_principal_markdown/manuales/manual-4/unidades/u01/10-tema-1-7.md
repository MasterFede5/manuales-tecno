# 1.7 Constrained output (JSON, schemas)

- **Output humano:** *"La categoría es inscripciones"* (inservible para código).
- **Output de máquina:** `{"categoria": "inscripciones"}` (consumible por código).
Aprenderás a forzar al modelo a hablar en código estructurado sin errores.

::concepto{titulo="Constrained output (salida restringida)"}
El **constrained output** obliga al modelo a generar respuestas bajo un **formato estructurado y predecible**.
- Usa habitualmente **JSON** basado en un *schema* estricto.
- Garantiza que la salida encaje perfectamente en tus bases de datos o APIs.

Existen 4 niveles de garantía para lograrlo:
- **Prompt + ejemplo** — pedir "responde en JSON con esta forma".
- **JSON mode** — flag del proveedor que garantiza JSON válido.
- **Tool use / Schema** — el modelo llena un schema definido.
- **Structured Outputs** — modelo sigue el schema con 100 % de validez.
::/concepto

::interioriza{titulo="Baja curva: Los moldes de galletas"}
- **Texto libre (sin schema):** Masa cruda que se desparrama sin forma.
- **JSON Schema:** Un molde metálico de galletas.
Al aplicar el molde, obligas a la IA a recortar su respuesta exactamente con los campos requeridos.
Nada de plastas de texto: solo datos estructurados y listos para servir.
::/interioriza

## JSON Schema como contrato

Un schema describe forma, tipos y valores permitidos:

```json
{
  "type": "object",
  "properties": {
    "categoria": { "type": "string", "enum": ["inscripciones", "queja", "otro"] },
    "confianza": { "type": "string", "enum": ["alta", "media", "baja"] },
    "requiere_humano": { "type": "boolean" }
  },
  "required": ["categoria", "confianza", "requiere_humano"],
  "additionalProperties": false
}
```

Pasas este schema al modelo vía `response_format` o `tools`.
El proveedor **garantiza** que la salida cumplirá este contrato al pie de la letra.

## Patrón de prompt para JSON

Tener un schema fuerte no te exime de dar contexto. Debes **describir cada campo** en el prompt:
- Explica qué significa cada *enum* de tu modelo de datos.
- Aclara bajo qué condiciones exactas un campo booleano debe ser `true`.

```xml
<task>Clasifica la solicitud y devuelve JSON conforme al schema.</task>

<schema_documentation>
- categoria: tema principal de la solicitud.
- confianza: tu certeza (alta, media, baja).
- requiere_humano: true si la situación es grave.
</schema_documentation>

<input>{texto}</input>
```

## Mecanismos por proveedor

- **OpenAI:** `response_format: {type: "json_schema", strict: true}` (100 % de validez).
- **Anthropic:** Tool use con `input_schema` (99 %+ de validez).
- **Google Gemini:** `responseSchema` en la API (Alta validez).
- **Local (Ollama):** Outlines, Guidance (Validez matemática por gramática).

## Casos de uso típicos

- **Clasificación:** JSON con `enum` de categorías.
- **Extracción:** JSON con `nombre`, `email`, `tema`.
- **Acciones (n8n):** JSON con `accion`, `parametros`.
- **Metadata:** JSON dual para respuesta humana y datos de máquina.

## Errores y blindaje

- **Schema laxo:** Si usas puro `string`, no validas nada. Usa `enum` o `pattern`.
- **`additionalProperties: true`:** El modelo inventará campos no esperados. Forzalo a `false`.
- **Falta validación runtime:** Siempre usa un parser (Zod, Pydantic) al recibir los datos.
- **Anidamiento profundo:** Más de 4 niveles es innecesario y confuso. Aplana la estructura.

## Patrón "respuesta + metadata"

En un solo llamado, puedes obtener **dos outputs integrados**:
1. Texto amigable para el usuario final.
2. Metadatos estructurados para la lógica de negocio.

```json
{
  "respuesta_humana": "Las inscripciones abren el lunes a las 9:00.",
  "metadata": {
    "categoria": "inscripciones",
    "confianza": "alta",
    "requiere_humano": false
  }
}
```

- **Frontend:** Muestra únicamente el campo `respuesta_humana`.
- **Backend:** Lee `metadata` para ejecutar acciones automatizadas.

::pausa{titulo="💭 Deducción lógica (Active Recall)"}
- **Validación vs Semántica:** Si tu prompt no explica el enum "confianza", pero usas *Structured Outputs* al 100%, ¿el JSON será válido sintácticamente? ¿Será preciso?
- **Blindaje:** ¿Por qué es vital exigir `additionalProperties: false` al guardar los datos en una base SQL?
- **UX dual:** ¿Qué problema arquitectónico resuelves al dividir `respuesta_humana` y `metadata` dentro del mismo objeto JSON?
::/pausa
