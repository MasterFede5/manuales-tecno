# 1.4 Self-consistency y Tree-of-Thoughts

Le pides al modelo un "plan de inscripciones". Te entrega uno plausible.
Se lo pides de nuevo y entrega otro distinto.
¿Cuál llevas a la junta? Aquí tienes el método.

::concepto{titulo="Self-consistency y Tree-of-Thoughts (ToT)"}
Ambas técnicas extienden Chain-of-Thought (CoT) explorando múltiples rutas:

* **Self-consistency:**
  * Genera múltiples cadenas de razonamiento independientes.
  * Elige la respuesta por mayoría (votación).
  * Ideal para: respuestas únicas (cálculos, clasificación).
* **Tree-of-Thoughts (ToT):**
  * Explora un árbol de decisiones.
  * Evalúa ramas, descarta las pobres y profundiza en las mejores.
  * Ideal para: problemas abiertos (planeación, diseño).
::/concepto

::interioriza{titulo="Baja curva: El comité del Asistente"}
Imagina que el Asistente Institucional es un comité escolar:
* Una sola opinión para clasificar un correo puede ser sesgada (CoT simple).
* Si le pides que lo analice tres veces y vote (Self-consistency), filtras el ruido.

La decisión final, por mayoría, será mucho más estable y confiable.
::/interioriza

## Self-consistency en 3 pasos

Implementar esto es como pedir varias segundas opiniones:

* **Repetición:** Lanza el prompt con `temperature=0.7` (3 a 10 veces).
* **Extracción:** Saca la respuesta final de cada corrida.
* **Votación:** La respuesta más frecuente gana.

```text
Ejemplo de votación:
Corrida 1: B  |  Corrida 2: B  |  Corrida 3: A  |  Corrida 4: B

Resultado: Gana B (3/4 votos).
```

* **Desventaja:** Mayor costo por N llamadas.
* **Ventaja:** Reduce el error un 10-25% en lógica y matemáticas.

## Tree-of-Thoughts en 4 pasos

Es útil para problemas complejos que requieren exploración y descarte:

* **Descomponer:** Divide el problema en pasos.
* **Generar:** Propone 3-5 alternativas por paso.
* **Evaluar:** Asigna un puntaje a cada rama según criterios.
* **Buscar:** Profundiza en las mejores ramas y poda el resto.

Patrón típico en el Asistente:

```xml
<tree_of_thoughts>
Paso 1: Generar 3 estrategias para inscripciones.
Paso 2: Listar sub-tácticas para cada una.
Paso 3: Evaluar factibilidad y podar las peores.
Paso 4: Desarrollar cronograma de las ramas ganadoras.
</tree_of_thoughts>
```

* **Advertencia de Costo:** Es 10-50x más caro que un prompt normal.
* Úsalo solo cuando fallar sea más costoso que las llamadas al modelo.

## ¿Cuándo elijo cada una?

| Característica del problema | Self-consistency | Tree-of-Thoughts |
|---|---|---|
| Respuesta única correcta | ✓ | (overkill) |
| Múltiples soluciones aceptables | (no aplica) | ✓ |
| Cálculo o lógica | ✓ | (no necesario) |
| Diseño, planeación, escritura | (no resuelve) | ✓ |
| Presupuesto bajo | ✓ (3 corridas) | ✗ |
| Decisión de alto impacto | ✓ + auditoría | ✓ con criterios explícitos |

## Reglas operativas para tu Asistente

Aplica estas reglas para optimizar el gasto de tokens:

* **Default:** Una sola pasada para consultas casuales.
* **Self-consistency:** Para clasificar correos o tickets (votar 3 veces).
* **ToT:** Exclusivo para planes mensuales estratégicos.
* **Logueo:** Guarda siempre las N corridas para auditar divergencias.

::pausa{titulo="💭 Pausa para pensar"}
* **Deducción 1:** Si todas las cadenas comparten un error sistemático, Self-consistency falla. ¿Cuándo ocurriría esto?
* **Deducción 2:** ToT con profundidad 4 y 5 ramas: ¿cuántas llamadas genera? ¿Es viable en producción diaria?
* **Deducción 3:** Nombra un proceso del Asistente Institucional que justifique ToT y otro que no.
::/pausa
