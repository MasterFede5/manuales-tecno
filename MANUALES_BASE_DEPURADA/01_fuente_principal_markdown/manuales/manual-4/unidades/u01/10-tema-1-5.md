# 1.5 Prompt chaining

- Pedir todo en un solo prompt da resultados mediocres.
- ¿La solución? Usar prompts especializados encadenados.
- Pasamos de un modelo "todólogo" a una cocina por estaciones.

::concepto{titulo="Prompt chaining"}
- **Prompt chaining** divide una tarea compleja en prompts pequeños.
- El resultado (output) de uno es la entrada (input) del siguiente.
- Cada eslabón hace una sola cosa y la hace bien.

Existen tres patrones típicos:
- **Lineal:** P1 → P2 → P3 (transformación secuencial).
- **Ramificado:** Un prompt abre ramas paralelas que luego se fusionan.
- **Condicional:** Un prompt decide el siguiente paso (router).

Esto mejora la calidad, el control y facilita detectar errores.
::/concepto

::interioriza{titulo="La línea de ensamblaje (Baja Curva)"}
Imagina una fábrica de automóviles.
Si un solo mecánico arma el auto entero, tardará y cometerá errores.
Dividir el trabajo en estaciones (chasis, motor) da velocidad y precisión.
Tu Asistente Institucional funciona igual con prompts encadenados.
::/interioriza

## Anatomía de una cadena lineal

Veamos un pipeline típico para un comunicado oficial:
- **P1 - Extracción:** Texto crudo → JSON estructurado.
- **P2 - Estructura:** JSON → Outline de secciones.
- **P3 - Redacción:** Outline → Borrador del comunicado.

¿Y los pasos finales?
- **P4 - Revisión:** Borrador → Versión con tono corregido.
- **P5 - Formato:** Versión final → Archivo HTML o PDF.

Cada paso tiene su propia plantilla y evaluación.
Si algo falla, sabes exactamente en qué "estación" ocurrió el error.

## Paralelización y Enrutamiento

**Patrón Ramificado (Paralelización):**
- Ejecuta tareas independientes al mismo tiempo.
- Acelera procesos hasta tres veces más rápido.
- Ejemplo: extraer acuerdos, compromisos y fechas a la vez.

**Patrón Condicional (Router):**
- Un prompt clasifica primero la entrada del usuario.
- Luego, dirige el flujo (ej. quejas vs. inscripciones).
- Ahorra tokens y especializa la respuesta final.

## Reglas y Anti-patrones

**Reglas para cadenas robustas:**
- **Una responsabilidad:** Divide si un prompt hace dos cosas.
- **Contratos explícitos:** Define entradas y salidas (ej. texto → JSON).
- **Validador y logueo:** Registra todo y reintenta si falla.

**Anti-patrones a evitar:**
- **Cadenas largas:** Más de 7 pasos. Mejor combina o paraleliza.
- **Contexto innecesario:** Recuerda que cada token procesado cuesta dinero.
- **Falta de idempotencia:** El mismo input siempre debe dar el mismo output.

## Caso aplicado en tu Asistente

Para el "comunicado de suspensión" usamos distintos modelos:
- **Extracción:** Modelo rápido y barato (ej. Haiku/GPT-mini).
- **Redacción:** Modelo potente y grande (ej. Opus/GPT-4o).
- **Formato:** Código determinístico puro, sin usar IA.

Todo el proceso queda versionado y auditable paso a paso.

::pausa{titulo="💭 Deducciones lógicas"}
1. Si cambias 2 de 5 pasos al modelo más barato, ¿cómo impacta en la latencia total?
2. Si un eslabón falla el 5% de las veces, ¿qué impacto tiene en una cadena lineal de 6 pasos?
3. En el router condicional, ¿por qué es crítico que su "contrato" sea estricto?
::/pausa

> **Frase puente.** Cada eslabón necesita aprender el patrón con pocos ejemplos para no romperse. Es el corazón del **few-shot estratégico**. Subtema 1.6.
