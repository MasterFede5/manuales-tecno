# 5.5 Webhooks, APIs y triggers

Tu workflow es perfecto, pero ¿quién lo arranca?
Si esperas a que alguien le dé "play", es asistencia, no automatización.
A continuación, los 3 mecanismos que disparan workflows sin humanos.

::concepto{titulo="Triggers, webhooks y APIs"}
**Trigger** es el evento que inicia un workflow. Hay tres familias:
- **Schedule:** Corre en horario fijo (cada 5 min, los lunes a las 8:00).
- **Polling:** Consulta periódica a una API ("¿hay correo nuevo?").
- **Webhook:** Sistema externo **notifica al workflow** cuando algo ocurre.

Una **API** es la interfaz programable de un servicio externo.
Es el estándar para consultar o modificar datos en otro sistema.
Ejemplos: Notion API, OpenAI API, Calendar API.
::/concepto

::interioriza{titulo="Baja Curva: El oído del workflow"}
Imagina que esperas un paquete importante en tu casa.
- **Schedule:** Sales a mirar el buzón cada 5 minutos.
- **Polling:** Llamas a la oficina de correos cada hora para preguntar.
- **Webhook:** El cartero toca el timbre cuando llega.
Elegir el timbre (Webhook) ahorra energía, dinero y muchas molestias.
::/interioriza

## Cuándo usar cada uno

- **Schedule:** Reporte semanal (lunes 8:00) o Backup de Notion (semanal).
- **Polling:** Detectar correo nuevo si la app no soporta webhooks.
- **Webhook:** Solicitud web, sincronización de Calendly, mención en Slack.

## Webhook y API 101

Un webhook es **una URL pública** de tu workflow.
El sistema externo hace `POST` con un payload JSON para ejecutarlo.

**Seguridad mínima del webhook:**
- URL larga, impredecible (generada con UUID).
- Autenticación con header `Authorization: Bearer <token>`.
- Rate limiting propio y HTTPS obligatorio.
- Validación de payload antes de procesar.

Cuando llamas a una **API externa**, recuerda:
- **Métodos HTTP:** GET (leer), POST (crear), PUT/PATCH (actualizar), DELETE.
- **Status codes:** 200 OK, 201 Created, 400 Bad, 401 Auth, 429 Rate Limit.
- **Rate limits:** Cada API tiene máximos (ej. Notion: 3 req/s).

## Patrones y Anti-patrones

**Patrones útiles:**
- **Webhook + ack inmediato:** Responde rápido y procesa async.
- **Polling con backoff:** Si no hay cambios, multiplica el intervalo.
- **Webhook con retry queue:** Si falla, encola y reintenta después.

**Anti-patrones:**
- **Schedule de 1 minuto:** Derroche costoso; usa polling con backoff.
- **Webhook sin auth:** Riesgo de que vacíen tu presupuesto.
- **Sin idempotencia:** Evita crear dos tickets por el mismo webhook.
- **Ignorar rate limit (429):** Tu workflow caerá; configura retry.

## Caso: Workflow Institucional

```text
[WEBHOOK] /admisiones (URL secreta, Bearer)
   ↓
[VALIDAR payload: nombre, email] → [VERIFICAR idempotency]
   ↓
[CALL Anthropic API] → [CALL Notion API] (retry on 429)
   ↓
[CALL Gmail API] → [CALL Slack API]
```
