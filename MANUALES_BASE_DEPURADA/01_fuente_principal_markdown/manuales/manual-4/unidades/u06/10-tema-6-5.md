# 6.5 Tool use / function calling (concepto)

Function calling es **el mecanismo técnico** central de los agentes.
- Sin él, solo hay charla (no hay loop ReAct ni acción real).
- Permite que el modelo interactúe con sistemas externos.
- Aprenderás la base conceptual sin escribir código complejo.

::concepto{titulo="Function calling / Tool use"}
**Function calling** o **Tool use** es la capacidad de un LLM de emitir un output estructurado (JSON).
- Indica qué función debe ejecutarse.
- Provee los argumentos necesarios para esa función.

El flujo es simple:
1. Envías el prompt + schemas JSON de las tools.
2. El modelo responde con una **tool call** (ej. `create_ticket`).
3. Tu sistema ejecuta la función real.
4. El resultado regresa al modelo en otro mensaje.
5. El modelo responde al usuario o llama otra tool.
::/concepto

::interioriza{titulo="El control remoto universal (Analogía)"}
Imagina un LLM sin *tool use* como un copiloto ciego que solo puede darte sugerencias verbales ("deberías encender la luz").
Con *tool use*, le das un control remoto (el JSON estructurado).
Ahora, él mismo presiona el botón `encender_luz(intensidad=50)`. 
Así, pasa de ser un bot conversacional a un operador en sistemas de producción.
::/interioriza

## El ciclo de tool use en detalle

```json
PASO 1 — Tu app envía request al modelo:
{
  "messages": [{"role": "user", "content": "Crea ticket para Andrea"}],
  "tools": [{
    "name": "create_notion_ticket",
    "description": "Crea fila en Notion DB Solicitudes...",
    "input_schema": { "type": "object", "properties": {...} }
  }]
}
```

```json
PASO 2 — El modelo decide usar tool:
{
  "content": [{
    "type": "tool_use", "name": "create_notion_ticket", "input": {...}
  }]
}
```

```python
PASO 3 — Tu código ejecuta la función:
result = notion.create_page(database_id="...", properties={...})
```

```json
PASO 4 y 5 — Mandas resultado y el modelo responde:
{
  "messages": [{"type": "tool_result", "content": "Creado: url=..."}]
}
-> "Ticket creado. URL: https://notion.so/..."
```

## El schema: contrato modelo-tool

Cada tool tiene 3 partes críticas:
- **Nombre:** Único y descriptivo (ej. `create_notion_ticket`).
- **Descripción:** Explica qué hace. **El modelo lee esto para decidir**.
- **Input schema:** Define variables, tipos, enums y requeridos.

Descripciones importan mucho:
- ❌ *"Crea ticket"* (Muy vago, propenso a errores).
- ✅ *"Crea fila en DB 'Solicitudes 2026'. Usar solo si categoría es 'inscripcion'."*

## Buenas prácticas de tool design

- **Una responsabilidad:** Separa `create` de `update`.
- **Schemas estrictos:** Usa enums para evitar alucinaciones.
- **Descripciones explícitas:** Pon contraejemplos claros.
- **Idempotencia:** Evita duplicados si el modelo llama la función dos veces.
- **Timeouts:** Un agente bloqueado por una API lenta es inútil.

## Multi-tool calling: paralelismo

Los modelos modernos pueden ejecutar **varias tools en paralelo**:
- Ejemplo: `search_kb()`, `get_profile()` y `check_calendar()`.
- Tu app ejecuta las tres a la vez y manda resultados juntos.
- Reduce drásticamente la latencia si no hay dependencia entre tareas.

## Tool choice forzada vs libre

Modos de ejecución de tools:
- **Auto:** (Default) El modelo decide si usa texto o tools.
- **Required:** Se obliga al modelo a usar **alguna** tool.
- **Forced:** Se obliga al modelo a usar **una tool específica** (ej: `classify`).

## Errores frecuentes

- **Exceso de tools (>30):** El modelo se confunde al elegir.
- **Schemas laxos:** El modelo inventa parámetros.
- **Descripciones basura:** Descripciones como "esta es la tool X".
- **Sin idempotencia:** Reintentos causan daños (ej: doble cobro).
- **Callar errores:** Si falla la API, debes enviar un `tool_error` al modelo.

## Tool use en plataformas no-code

Las plataformas abstraen los pasos complejos; tú solo creas schemas:
- **n8n:** Nodos del workflow expuestos como tools.
- **Dify:** Marketplace interno y custom tools.
- **OpenAI/Anthropic:** APIs directas muy maduras.
- **Lindy:** Tools predefinidas listas para usar.
