# 2.5 Spec-driven prompting

Tu spec dice "el Asistente clasifica solicitudes en 5 categorías". Tres semanas después, alguien añade una sexta categoría a la spec y olvida actualizar el prompt. El prompt sigue clasificando 5. **Producción miente sobre lo escrito.** Spec-driven prompting elimina esa grieta.

::concepto{titulo="Spec-driven prompting"}
**Spec-driven prompting** es la disciplina de **derivar el prompt directamente desde la especificación**, automatizando lo más posible la traducción spec → prompt para que ambos siempre estén sincronizados.

Tres formas concretas:
1. **Spec como prompt** — el prompt **es** la spec, escrita en lenguaje que el modelo entiende y un humano puede revisar.
2. **Generador spec→prompt** — un meta-prompt toma la spec estructurada y produce el prompt operativo.
3. **Spec ejecutable** — la spec contiene tanto la descripción como los ejemplos golden, y un script genera prompt + tests + dashboard.

El objetivo: una sola fuente de verdad. Si la spec cambia, el prompt cambia automáticamente.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Una receta escrita en una libreta y unos ingredientes en un mostrador. Si actualizas la receta pero no avisas al cocinero, va a salir el plato viejo. Si tu cocinero **lee la receta cada vez antes de cocinar**, no hay desfase. Spec-driven prompting hace que tu prompt "lea la spec" cada vez antes de servir.
::/interioriza

## Forma 1 — La spec ES el prompt

Caso simple, pero potente. La spec se escribe directamente como un prompt XML estructurado y vive en el repo. No hay traducción: la "documentación" y el "código" son el mismo archivo.

```xml
<!-- specs/clasificador-solicitudes.xml -->
<role>
Eres el clasificador del Asistente Institucional Albatros.
</role>

<categories>
- inscripciones: dudas o solicitudes sobre admisiones, fechas, requisitos.
- calificaciones: temas de notas, boletas, revisiones.
- becas: descuentos, apoyos, requisitos de beca.
- queja: malestar manifiesto del usuario que requiere de-escalación.
- otro: cualquier tema fuera de las 4 anteriores.
</categories>

<rules>
- Si la solicitud toca dos categorías, prioriza la más específica.
- Si hay tono molesto, marcalo aunque la categoría sea otra.
- Si la confianza es baja, marca requiere_humano: true.
</rules>

<output>JSON conforme al schema en schemas/categoria.json</output>
```

Lo legible para un humano de producto **es** lo ejecutable por la máquina. Cuando alguien actualiza la spec, está actualizando el prompt.

## Forma 2 — Generador spec→prompt

Cuando la spec es compleja (varias páginas), tienes un script que la consume y genera el prompt:

```yaml
# specs/clasificador.yaml
modulo: clasificador-solicitudes
proposito: Clasificar dudas en 5 categorías
entrada: texto libre del padre/estudiante
salida: JSON con categoria, confianza, requiere_humano
categorias: [inscripciones, calificaciones, becas, queja, otro]
reglas_de_priorizacion: ...
ejemplos_golden: ../evals/clasificador-v1.jsonl
modelo_recomendado: claude-haiku-4-5
```

Un script `generate.py` lee el YAML y produce `prompts/clasificador.v1.xml` + `tests/test_clasificador.py` + `monitor/dashboard.json`. **Toda la cadena se regenera al cambiar el YAML.**

## Forma 3 — Spec ejecutable

La forma más madura: la spec es un Markdown con frontmatter ejecutable. Cada bloque tiene tipo (descripción, regla, ejemplo, métrica). Un único comando `npm run build` produce todos los artefactos.

```markdown
---
modulo: redactor-comunicado
version: 1.0
---

# Redactor de Comunicados

## Descripción
Redacta comunicados oficiales para padres a partir de motivo + fecha + alcance.

## Reglas
1. Tono "usted", sin emojis.
2. Máximo 200 palabras.
3. Si falta dato → marcar [PENDIENTE].

## Ejemplos golden
::ejemplo
input: motivo=mantenimiento agua, fecha=2026-05-12
output: Estimadas familias: les comunicamos que el martes 12 de mayo...
::/ejemplo

## Métricas
- validez_schema: 100 %
- tono_correcto: ≥ 4/5 humano
```

## Por qué esta disciplina cambia el equipo

Antes: producto escribe spec en Notion, ingeniería traduce a prompt en otro repo, evaluación construye golden en otra hoja. Tres copias **siempre desincronizadas**.

Después: una spec, un comando, todo coherente. Cuando la dirección dice "agreguen una categoría", actualizas la spec, corres build, todo el sistema queda al día en 5 minutos.

## Anti-patrones del spec-driven

- **Generador con magia oscura.** Si nadie sabe qué hace `generate.py`, vuelves a la opacidad. Documenta.
- **Spec en lenguaje no procesable.** "El sistema debería ser justo" no genera prompt útil. Aterriza en reglas verificables.
- **Spec sin versión.** Sin semver de la spec, no sabes qué prompt corresponde a qué versión.
- **Generador que regenera todo cada vez.** En modelos pagos, esto puede salir caro. Diff selectivo: regenera solo lo que cambió.

## Recomendación práctica para tu Asistente

Empieza con **Forma 1 (spec ES el prompt)** para módulos simples (clasificador, extractor). Migra a **Forma 2** cuando tengas 3+ módulos y quieras consistencia. Reserva **Forma 3** para cuando tu equipo crezca y necesites onboarding rápido.

## Caso real: el día que dirección cambió la spec

Director: "agreguen una categoría 'salud' separada del resto, con escalación inmediata".

- 9:00 actualizas la spec (5 min): agregas categoría con regla `requiere_humano: true SIEMPRE`.
- 9:05 corres `npm run build`: regenera prompt, golden y dashboard.
- 9:10 corres `npm test`: golden pasa, schema OK.
- 9:15 abres PR con diff de 1 archivo conceptual; el equipo lo revisa.
- 10:00 deploy a canary 5 %. Sin sorpresas.

Sin spec-driven: 3 semanas de sincronización entre producto, eng y eval.
