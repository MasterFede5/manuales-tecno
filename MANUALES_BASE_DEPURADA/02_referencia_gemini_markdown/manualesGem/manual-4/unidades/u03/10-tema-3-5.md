# 3.5 Compartir y embeber

Tu Artifact funciona perfecto en tu pantalla. ¿Cómo lo ven los demás? Un link, un iframe, un download, un deploy real. Cada vía tiene sus reglas.

::concepto{titulo="Distribución de un Artifact"}
**Distribución** es el proceso de hacer que un Artifact pase de existir solo en tu chat a estar disponible para terceros. Cuatro vías:

1. **Link público (publish nativo)** — URL `claude.site/...` o `chat.openai.com/canvas/...`.
2. **Embed con iframe** — incrustar el Artifact en otro sitio (Notion, sitio institucional, intranet).
3. **Download** — descargar HTML/JSX y abrir local o subir a servidor propio.
4. **Deploy real** — exportar el código y desplegar en Vercel/Netlify/Cloudflare Pages.

Cada vía resuelve un problema distinto y tiene riesgos distintos.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Construir es la mitad. **Distribuir es lo que convierte tu mini-app en valor para alguien**. Si publicas mal, tu trabajo queda atrapado en tu computadora; si publicas con criterio, tu director enseña el dashboard del Asistente en una junta sin que tú estés presente.
::/interioriza

## Vía 1: Link público (lo más rápido)

**Cómo:** botón **Publish** en Claude Artifacts, **Share** en ChatGPT Canvas, **Share** en Gemini Canvas.

**Resultado:** URL `claude.site/abc123` (similar en otros). Cualquiera con el link entra.

**Pros:**
- Cero fricción.
- Cero costo.
- Cero infraestructura tuya.

**Contras:**
- **Sin autenticación.** Cualquiera con el link entra.
- **El proveedor puede deprecar la URL** o cambiar términos.
- **No es tuyo.** Si Anthropic cierra esa función, tu link muere.
- **Sin métricas.** No sabes cuántas personas entraron, desde dónde, cuánto tiempo.

**Cuándo usar:** demos, prototipos, portafolio personal, contenido público no sensible.

## Vía 2: Embed con iframe

Algunos proveedores permiten que copies código `<iframe>` y lo pegues en Notion, en tu sitio o en una intranet.

```html
<iframe src="https://claude.site/abc123/embed"
        width="100%" height="600"
        frameborder="0"></iframe>
```

**Pros:**
- El Artifact aparece dentro de **tu** página, no en otra.
- El usuario "no sabe" que hay un Artifact detrás.

**Contras:**
- Dependes igual del proveedor.
- Mismo problema de autenticación (la URL embebida sigue siendo pública).
- Limitaciones de iframe (no se puede llamar APIs del padre, etc.).

**Cuándo usar:** integrar dashboards en Notion, intranets internas, sitios estáticos.

## Vía 3: Download (saca el código y vuela)

Botón **Download** o copiar manualmente el código del panel.

**Pros:**
- **Es tuyo.** Si el proveedor desaparece, tu código sigue.
- **Auditable.** Lees y modificas a mano.
- **Sin dependencia.**

**Contras:**
- Tienes que **alojarlo** en algún lado para que otros lo abran.
- Sin URL nativa para compartir.

**Cuándo usar:** cuando quieres independencia del proveedor o cuando vas a desplegar a producción.

## Vía 4: Deploy real (la más adulta)

Pasos típicos:

1. Descarga código del Artifact (Code tab → Copy/Download).
2. Crea proyecto Vite + React local (`npm create vite@latest`).
3. Pega el código del Artifact en `App.jsx`.
4. Instala dependencias (`npm install recharts lucide-react`).
5. `git init` + push a GitHub.
6. Conecta GitHub a Vercel (https://vercel.com) o Cloudflare Pages.
7. Deploy automático.

**Resultado:** URL `tudominio.vercel.app` o `tudashboard.albatros.edu.mx`.

**Pros:**
- Dominio propio.
- Auth real (Clerk, Auth0, Supabase Auth).
- Bases de datos reales (Supabase, Firebase).
- Métricas de uso (Vercel Analytics, Plausible).
- CI/CD (cada push redeploya).

**Contras:**
- Configuración inicial (~2 horas la primera vez).
- Costo si supera tier gratis (Vercel free hasta cierto tráfico).
- Mantenimiento: dependencias, parches, runtime.

**Cuándo usar:** producción real, tráfico esperado, datos sensibles, integración con sistemas internos.

## Riesgos comunes y blindaje

- **Subir Artifact con datos personales.** Aunque sea borrador, la URL pública existe. **Anonimiza siempre** antes de Publish.
- **Asumir que la URL pública es eterna.** Configura backup: descarga el código en cuanto el Artifact funcione.
- **Compartir en redes sin pensar.** Un link público en Twitter puede generar carga inesperada o filtrar info de stakeholders.
- **Embed en sitio público con datos institucionales.** Si la URL está embebida en un sitio público, los datos también.

## Patrón recomendado para tu Asistente

| Etapa | Vía recomendada |
|---|---|
| Demo a director (ad-hoc) | Link público |
| Dashboard interno equipo | Embed en Notion del equipo |
| Backup propio | Download + commit en repo |
| Producción con login institucional | Deploy real en Vercel + Auth |

## Caso aplicado: el flujo de tu equipo

1. **Lunes 8:00** — actualizas dashboard con métricas de la semana en Claude.
2. **Lunes 8:15** — botón Publish, copias URL.
3. **Lunes 8:16** — pegas en canal Slack #albatros-asistente.
4. **Lunes 8:20** — descargas código y haces commit en `tools/dashboard/v_2026-04-28.jsx`.
5. **Cuando llegue el momento de producción real** (cuando dirección lo pida), desplegamos a Vercel con Clerk auth.
