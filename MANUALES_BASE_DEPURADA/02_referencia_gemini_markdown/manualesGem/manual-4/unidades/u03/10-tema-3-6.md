# 3.6 Versionado y control de cambios

Hoy es martes y tu dashboard funciona. El miércoles iteras 8 veces y termina mejor. El jueves iteras 5 veces y termina **peor**. ¿Cómo recuperas el bueno del miércoles? Sin versionado, lloras. Con versionado, dos clicks.

::concepto{titulo="Versionado de Artifacts"}
El **versionado de Artifacts** combina dos capas:

1. **Versionado interno del proveedor** — Claude/ChatGPT/Gemini guardan automáticamente cada iteración como v1, v2, v3 navegables dentro del chat.
2. **Versionado externo en repo** — descargas el código y lo guardas en Git con etiquetas semánticas (semver), CHANGELOG y commits descriptivos.

La capa 1 es cómoda pero frágil (depende del proveedor). La capa 2 es duradera. Equipos serios usan **ambas en tándem**.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Cuando vas al cine y la película no te gusta, te paras y te vas. Cuando construyes un dashboard, no puedes "irte": tu equipo ya lo está usando. **Necesitas regresar a la versión que funcionaba** sin perder los pedazos buenos del intento fallido. Esa es la promesa del versionado.
::/interioriza

## Capa 1: versiones internas del proveedor

En Claude Artifacts ves un selector v1, v2, v3, v4 con flechas ←/→. Cada vez que el modelo edita el artifact, se crea una versión.

**Operaciones útiles:**
- **Restaurar versión previa** — flechas o click en versión anterior. El siguiente edit parte de ahí.
- **Comparar visualmente** — abres en pestañas separadas v3 y v5 para ver diferencias.
- **Bifurcar** — restauras v3 y empiezas una nueva línea de iteración (v3 → v4'). Las antiguas se conservan.

**Riesgos:**
- El proveedor puede limitar historial (Claude conserva las últimas N).
- Si cierras el chat y vuelves, no siempre todas las versiones quedan.
- No tienes diff textual; comparas a ojo.

## Capa 2: versionado externo (Git)

Cada vez que llegas a una versión "estable" (lista para mostrar a alguien), descargas y commiteas:

```bash
# Estructura sugerida
mi-asistente/
├── tools/
│   └── dashboard/
│       ├── v0.1.0/
│       │   ├── App.jsx
│       │   ├── package.json
│       │   └── snapshot.png
│       ├── v0.2.0/
│       ├── v1.0.0/
│       └── CHANGELOG.md
```

**Convenciones útiles:**
- **Semver** (v0.1.0, v0.2.0, v1.0.0): mayor para cambios breaking, menor para features, patch para fixes.
- **CHANGELOG** en lenguaje humano: *"v0.2.0 agrega tab de fuentes y filtro por módulo. Breaking: cambia campo `fechas` de string a array."*
- **Snapshot** (captura de pantalla) por versión: ayuda a recordar qué fue cada una sin abrir código.
- **Commit message convencional**: `feat: agrega tab de fuentes`, `fix: corrige cálculo de tasa`, `chore: actualiza datos de ejemplo`.

## Política mínima del equipo (1 página, 1 hora de armar)

```markdown
# Política de versionado de Artifacts — Asistente Albatros

## Cuándo "publicamos" una versión
- Cuando un stakeholder externo verá el Artifact (junta, demo, link compartido).
- Cuando el contenido entró a producción real (deployed).
- Antes de empezar una refactorización mayor.

## Qué hacemos al publicar
1. Descargar código desde el panel del proveedor.
2. Crear carpeta `tools/<nombre>/v<X.Y.Z>/`.
3. Tomar captura del Artifact y guardarla como `snapshot.png`.
4. Actualizar `CHANGELOG.md` con resumen de cambios.
5. Commit en Git con mensaje convencional.
6. Tag: `git tag v0.2.0 && git push --tags`.

## Restauración
- Versión interna (Claude): selector de versión.
- Versión externa (Git): `git checkout v0.1.0`.
```

## Anti-patrones que cuestan caro

- **Solo confiar en versiones internas.** El día que el proveedor cambia política o se cae, pierdes meses.
- **Descargar sin etiquetar.** Tres archivos `App.jsx`, `App-final.jsx`, `App-final-v2-FIX.jsx` y nadie sabe cuál es cuál.
- **CHANGELOG vacío.** Versionas pero nadie sabe qué cambió. Inutilizable después de 4 versiones.
- **Snapshot inexistente.** Sin captura visual, no recuerdas a qué se veía v0.3.0.
- **Versionar todo, todo el tiempo.** Si commiteas cada iteración menor, el repo es ruido. Versiona solo "estables".

## Caso real: la regresión del jueves

Martes — v1.0 publicada al equipo. Funciona.
Miércoles — v1.1 con tab nuevo, mejora.
Jueves — v1.2 intenta agregar dark mode, **rompe responsividad mobile**.
Viernes — alguien abre en su celular el link y reporta el bug.

Sin versionado externo: 2 horas de debug intentando recordar qué cambió.
Con versionado: `git checkout v1.1` (30 segundos), redeploy (2 min), problema resuelto. Diferencia: **45 minutos de equipo salvados**, además de la confianza de stakeholders.

## Tres reglas de oro

1. **Versiona al publicar.** Si vas a mostrar a alguien, hay versión.
2. **Snapshot siempre.** Captura de pantalla > 0 lecturas de código.
3. **CHANGELOG en humano.** Si lo escribes para tu yo de 6 meses, está bien.
