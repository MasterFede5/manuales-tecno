# 3.2 Claude Artifacts (creación, iteración, publicación)

Abres claude.ai, escribes "construye un dashboard simple para mi Asistente Institucional con 4 métricas, una lista de módulos y un botón de exportar a CSV". Treinta segundos después, el panel derecho muestra un dashboard funcional. Pides "que el botón sea naranja" y, sin reescribir nada, el modelo edita y vuelves a verlo en vivo. Esto es Claude Artifacts.

::concepto{titulo="Claude Artifacts"}
**Claude Artifacts** es el sistema de Anthropic para crear, iterar y publicar piezas autocontenidas (documentos, código, mini-apps web) en un panel paralelo al chat. Disponible en `claude.ai` (gratis con límites; sin límite con Pro/Team), soporta nativamente:

- **React** con Tailwind y librerías curadas (`recharts`, `lucide-react`, `framer-motion`, etc.).
- **HTML/CSS/JS** estático.
- **SVG**, **Mermaid**, **Markdown**.
- **Código** en cualquier lenguaje (no ejecutable, pero formateado y descargable).

Lanzado en junio 2024 y consolidado en 2025 como la referencia para "vibe coding" y prototipado conversacional.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Antes de Artifacts, llevar una idea de "se me ocurre X" a "tengo un demo navegable" tomaba 2 horas mínimo: scaffold, deploy, iterar, redeploy. Hoy son 5 minutos en una sola conversación. La velocidad no es solo cómoda — **cambia el tipo de ideas que te atreves a probar**. La fricción mata la curiosidad; Artifacts la libera.
::/interioriza

## Anatomía de la interfaz

```
┌─────────────────────────┬───────────────────────────────┐
│                         │  ┌─[Code][Preview]─[v3]─[⋯]┐ │
│  CHAT                   │  │                          │ │
│  ─────                  │  │       ARTIFACT           │ │
│  > "haz un dashboard"   │  │      (vista preview)     │ │
│  Claude: aquí está...   │  │                          │ │
│                         │  │                          │ │
│  > "agrega tab de       │  │                          │ │
│     fuentes"            │  │                          │ │
│  Claude: actualizado.   │  │                          │ │
│                         │  └──────────────────────────┘ │
│  [campo de texto]       │  [Publish] [Copy] [Download]  │
└─────────────────────────┴───────────────────────────────┘
```

Lado izquierdo: conversación. Lado derecho: el artifact con dos vistas (Code/Preview), navegación por versiones (v1, v2, v3, …) y acciones (Publish, Copy, Download).

## Patrón de creación: el "primer prompt"

El primer prompt del artifact define la calidad del resto. Estructura recomendada:

```
Crea un Artifact en [React | HTML | SVG | Mermaid | Markdown] que muestre:

PROPÓSITO: [una frase]
COMPONENTES: [lista numerada]
DATOS DE EJEMPLO: [json o tabla embebida]
ESTILO: [paleta, tipografía, mood]
INTERACCIONES: [qué pasa al click/hover]
RESTRICCIONES: [tamaño, mobile-first, sin libs externas a las soportadas]
```

Ejemplo aplicado a tu Asistente:

```
Crea un Artifact en React que muestre el dashboard del Asistente Institucional Albatros.

PROPÓSITO: Una página que un stakeholder no técnico pueda leer en 60 segundos.
COMPONENTES:
1) Header con logo Albatros y status global (verde/amarillo/rojo).
2) 4 métric-cards: solicitudes resueltas, tasa de escalación, latencia mediana, costo del mes.
3) Tabla de módulos: clasificador, redactor, resumen, cada uno con badge de estado.
4) Tab "Decisiones" con 3 ADRs.
5) Tab "Riesgos" con 2 alertas.

DATOS DE EJEMPLO: incluye un objeto con valores plausibles para que la preview no esté vacía.
ESTILO: Tailwind, paleta azul Albatros (#0E3A8A) y acento naranja (#F39C12), tipografía Inter, sombras suaves.
INTERACCIONES: tabs cambian sin recargar; cards muestran tendencia con sparkline.
RESTRICCIONES: una sola pantalla, mobile-first, solo libs soportadas (recharts, lucide-react).
```

Este prompt produce un dashboard usable en la primera pasada.

## Iteración fluida (el corazón del valor)

Una vez creado, las iteraciones son frases naturales:

- *"Cambia el badge a rojo cuando latencia > 3 s."*
- *"Agrega un input para buscar en la tabla de módulos."*
- *"Mueve la métrica de costo a la derecha y agrégale tooltip con desglose."*
- *"Tira esto y empieza de cero con un layout de 2 columnas."*

El modelo edita el artifact en su lugar. Si una iteración te queda mal, **vuelves a la versión previa** con el selector v1/v2/v3.

## Cómo Claude decide cuándo crear un Artifact

Reglas internas (documentadas por Anthropic):

- **Texto >800 palabras** y editable.
- **Código >15 líneas** que tiene sentido reusar.
- **HTML/CSS/SVG/Mermaid** explícitamente solicitados.
- **React** cuando hay interactividad.
- **Visual / autocontenido** que se beneficia de preview.

No crea Artifact si:
- Tu pregunta es exploratoria (vas a iterar mucho la idea, no la forma).
- El contenido cambia cada turno (no se beneficia de versión).
- Es información sensible (Claude favorece inline para tu privacidad).

Puedes forzar: *"créalo como Artifact"* o *"no lo pongas en Artifact, dámelo inline"*.

## Publicación

Botón **Publish** genera URL pública en `claude.site/...`. Cualquiera con el link lo abre en su navegador. **Limitaciones**:

- Sin autenticación nativa (cualquiera con link entra).
- Sin estado persistente compartido entre visitas (cada visita arranca de cero).
- El proveedor puede deprecar URLs (lee TOS).
- No es producción: es **demo o portafolio**.

Para producción real necesitas exportar (botón Download, copiar HTML/JSX) y desplegarlo tú mismo en Vercel, Netlify, Cloudflare Pages.

## Remix: aprovechar artifacts ajenos

Cuando ves un Artifact publicado, en versiones recientes hay botón **Remix** que abre el código en tu chat para modificarlo. Es la versión Claude del *fork* en GitHub. Útil para:
- Aprender de patrones bien hechos.
- Construir sobre prototipos compartidos.
- No partir de cero cuando alguien ya resolvió un caso similar.

## 5 prácticas que separan un Artifact pobre de uno excelente

1. **Primer prompt completo y claro.** Lleva más tiempo que improvisar, ahorra 10 iteraciones.
2. **Datos de ejemplo plausibles.** No "lorem ipsum"; valores realistas para tu caso.
3. **Iteraciones pequeñas.** Una mejora por mensaje. Si pides 5 cambios juntos, alguno saldrá mal.
4. **Vuelve a versión previa cuando algo empeora.** No "lo arregles encima"; restaura.
5. **Cierra con polish pass.** Último prompt: *"Haz una pasada final: revisa accesibilidad, responsividad, errores en consola, ortografía."*

## Errores frecuentes y blindaje

- **Pedir "todo en uno" desde el primer prompt.** Mejor descomponer: layout primero, luego datos, luego interacciones, luego polish.
- **No especificar paleta o tipografía.** El modelo elige por defecto; queda genérico.
- **Olvidar mobile.** Decir "mobile-first" en el prompt cambia el resultado.
- **Confiar en estado persistente.** No lo hay; usa props o localStorage.
- **Querer fetch a APIs externas.** No funciona en sandbox; pasa datos hardcoded o usa export + deploy real.

## Caso aplicado: tu dashboard semanal

Cada lunes, tu equipo actualiza un JSON con métricas de la semana. Cargas el JSON al chat junto al prompt del dashboard, Claude lo embebe en el Artifact y publicas el link al canal de Slack del equipo. Tiempo total semanal: 8 minutos.
