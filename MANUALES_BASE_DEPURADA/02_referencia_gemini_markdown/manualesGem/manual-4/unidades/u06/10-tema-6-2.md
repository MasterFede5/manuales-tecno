# 6.2 Anatomía: percepción, planificación, herramientas, memoria

Un agente que parece "inteligente" tiene siempre los mismos 5 órganos. Si falta uno, el resto no compensa. Esta lección los disecciona uno por uno con ejemplos del Asistente Institucional.

::concepto{titulo="Anatomía de un agente"}
Un agente robusto tiene 5 componentes integrados:

1. **Percepción** — cómo recibe el input del mundo: texto, imagen, audio, eventos.
2. **Planificación** — cómo descompone el objetivo en sub-tareas y decide el siguiente paso.
3. **Herramientas (tools)** — el conjunto de funciones invocables que tiene a disposición.
4. **Memoria** — qué recuerda entre pasos, conversaciones y sesiones.
5. **Ejecución** — el motor que coordina todo: hace llamadas, controla loop, aplica safeguards.

Ningún componente se construye en aislado: la percepción debe alimentar planificación, la planificación elige tools, los tools modifican memoria, y la ejecución gobierna el ritmo y los límites.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Cuando una empresa de auto contrata a un piloto, no le pone solo motor: pone ojos (percepción), navegador (planificación), volante (tools), bitácora (memoria) y conductor (ejecución). Falta uno y el viaje se interrumpe. Tu agente es exactamente eso: un sistema completo, no un LLM con APIs colgadas.
::/interioriza

## Componente 1 — Percepción

**Cómo recibe el agente la información del mundo.**

Modalidades:
- **Texto** (chat, email, ticket, doc).
- **Imagen** (multimodal: GPT-4o, Claude Opus, Gemini).
- **Audio** (transcripción Whisper, AssemblyAI).
- **Eventos del sistema** (webhook recibido, schedule disparado, archivo subido).
- **Estado de UI** (computer use: el agente ve la pantalla).

Práctica institucional: el Asistente recibe (a) texto del padre, (b) opcional captura del problema, (c) eventos de sistemas que escucha (correo nuevo en admisiones).

## Componente 2 — Planificación

**Cómo el agente descompone el objetivo en pasos.**

Estilos:
- **Reactivo** — sin plan explícito, decide cada paso basado en último estado (CoT corto).
- **Plan-and-execute** — primero genera plan completo (lista de pasos), luego ejecuta.
- **ReAct** — alterna Thought-Action-Observation (cubierto en U1).
- **Reflexion** — añade autoevaluación tras cada paso para corregir.
- **Tree-of-Thoughts** (U1) — explora múltiples planes alternativos.

Para el Asistente, plan-and-execute con verificación es lo más estable. Para tareas creativas, ReAct con reflexion.

## Componente 3 — Herramientas (tools)

**Lo que el agente puede hacer en el mundo.**

Categorías:
- **Información** — search web, RAG sobre tu corpus, query DB.
- **Acción** — crear/modificar registros, mandar email, postear, agendar.
- **Cómputo** — code interpreter, calculadora, transformaciones.
- **Comunicación** — postear Slack, llamar API humana.
- **Observación** — capturar pantalla, leer status.

Cada tool tiene **schema** (qué espera, qué devuelve), **descripción** (cuándo usarla) y **permisos** (quién puede ejecutarla).

```json
{
  "name": "create_notion_page",
  "description": "Crear página nueva en Notion DB de admisiones cuando recibes solicitud nueva",
  "input_schema": {
    "type": "object",
    "properties": {
      "nombre": {"type": "string"},
      "email": {"type": "string", "format": "email"},
      "categoria": {"enum": ["inscripcion", "info", "queja"]}
    },
    "required": ["nombre", "email", "categoria"]
  }
}
```

Tu Asistente Institucional comienza con 6-12 tools curadas. Más es ruido para el agente.

## Componente 4 — Memoria

**Qué persiste entre pasos, conversaciones y sesiones.**

Capas:
- **Working memory** — contexto inmediato del modelo (contexto window).
- **Episodic memory** — recuerdo de la conversación actual (historial).
- **Semantic memory** — conocimiento estructurado (RAG, BD).
- **Procedural memory** — patrones aprendidos (few-shot, ejemplos).
- **Long-term memory** — datos persistentes (vector store dedicado al usuario, perfil).

Frameworks como **Mem0**, **Letta**, **MemGPT** especializan en memoria de agentes. Para 90 % de los casos institucionales, basta con: contexto window + RAG + perfil de usuario en BD.

## Componente 5 — Ejecución (orquestador)

**El motor que conduce todo el loop.**

Responsabilidades:
- Recibir percepción → llamar a planificación.
- Tomar decisión del LLM → invocar tool correcto.
- Capturar resultado → alimentar siguiente paso.
- Aplicar **límites**: max_iterations, max_tokens, max_cost.
- **Logging y observabilidad** de cada paso.
- **Manejo de errores**: retry, fallback, escalar a humano.
- **Approval loops** para acciones críticas.

Plataformas: LangGraph, CrewAI, AutoGen, OpenAI Assistants API, Anthropic Tool Use con loops, n8n AI Agent node.

## Patrón de implementación mínimo

```python
# pseudocódigo conceptual
agent = create_agent(
    model="claude-opus-4-7",
    tools=[search_kb, create_notion, send_email, schedule_calendar],
    memory=working_mem + rag_mem,
    max_iterations=10,
    max_cost_usd=2.0,
    on_tool_call=log_step,
    on_error=alert_humanly
)

result = agent.run("Procesa cambio de turno de Andrea Hernández")
```

Las plataformas no-code (n8n AI Agent, Dify, Stack AI) exponen lo mismo con UI gráfica.

## Errores frecuentes

- **Faltar uno de los 5.** Un agente sin memoria olvida el paso anterior; sin límites, gasta sin control.
- **Tools mal descritos.** Si la descripción es ambigua, el agente elige el tool incorrecto.
- **Demasiados tools.** 30+ tools confunden al modelo. Cura.
- **Memoria sin TTL.** Si guardas todo, el contexto se llena de basura.
- **Sin observabilidad.** Imposible debug sin logs de cada Thought-Action-Observation.
