# 6.5 Tool use / function calling (concepto)

Function calling es **el mecanismo técnico** que habilita agentes. Sin él, no hay loop ReAct ni acción real: solo charla. Esta lección te lo explica sin que escribas código, pero entendiendo lo suficiente para diseñar bien.

::concepto{titulo="Function calling / Tool use"}
**Function calling** (OpenAI) o **Tool use** (Anthropic, Google) es la capacidad de un LLM de **emitir un output estructurado que indica qué función externa debe ejecutarse y con qué argumentos**, en lugar de o además de texto.

El flujo:
1. Tú envías al modelo: prompt + lista de tools disponibles (cada uno con schema JSON).
2. El modelo decide si responde con texto o con una **tool call** (`{"name": "create_ticket", "arguments": {...}}`).
3. Tu código (o la plataforma) ejecuta la función.
4. El resultado regresa al modelo en una nueva llamada.
5. El modelo decide responder al usuario o llamar otro tool.

Es la base sobre la que se construyen ReAct, agentes, y todas las plataformas de la sección 6.4.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Antes de function calling (2023), si querías que un modelo "consultara la base de datos", tenías que parsear su respuesta de texto con regex frágil ("¿dijo SELECT? ¿la query es correcta?") y rezar. Function calling te entrega **un JSON limpio listo para ejecutar**. La diferencia: el agente pasa de un experimento de laboratorio a producción en sistemas serios.
::/interioriza

## El ciclo de tool use en detalle

```
PASO 1 — Tu app envía request al modelo:
{
  "model": "claude-opus-4-7",
  "messages": [
    {"role": "user", "content": "Crea ticket de inscripción para Andrea Hernández"}
  ],
  "tools": [
    {
      "name": "create_notion_ticket",
      "description": "Crea fila en Notion DB Solicitudes con datos del aspirante",
      "input_schema": {
        "type": "object",
        "properties": {
          "nombre": {"type": "string"},
          "email": {"type": "string"},
          "categoria": {"enum": ["inscripcion", "info", "queja"]}
        },
        "required": ["nombre", "email", "categoria"]
      }
    },
    { ... otros tools ... }
  ]
}

PASO 2 — El modelo responde con tool_use:
{
  "stop_reason": "tool_use",
  "content": [
    {"type": "tool_use", "id": "tu-1", "name": "create_notion_ticket",
     "input": {"nombre": "Andrea Hernández", "email": "andrea.h@example.com", "categoria": "inscripcion"}}
  ]
}

PASO 3 — Tu código ejecuta la función real:
result = notion_client.create_page(database_id="...", properties={...})
# result = {"page_id": "abc123", "url": "https://notion.so/..."}

PASO 4 — Mandas el resultado al modelo:
{
  "messages": [
    ... mensajes previos ...,
    {"role": "user", "content": [
      {"type": "tool_result", "tool_use_id": "tu-1",
       "content": "Creado: page_id=abc123, url=https://notion.so/..."}
    ]}
  ]
}

PASO 5 — El modelo responde al usuario:
"Ticket creado para Andrea Hernández. Folio asignado: ABC-123. URL: https://notion.so/..."
```

## El schema: el contrato entre modelo y tool

Cada tool tiene 3 partes:

1. **Nombre** — único, descriptivo (`create_notion_ticket`, no `tool1`).
2. **Descripción** — cuándo usarlo, qué hace. **Esto es lo que el modelo lee para decidir**. Si la descripción es ambigua, el modelo elige mal.
3. **Input schema** — JSON Schema con tipos, enums, required, format.

Ejemplo de descripción **mala** vs **buena**:
- ❌ *"Crea ticket"* — vago.
- ✅ *"Crea una nueva fila en la base de datos Notion 'Solicitudes admisión 2026' cuando un aspirante envía una solicitud nueva. Usar solo cuando la categoría es 'inscripcion' y se tienen al menos nombre y email del solicitante. NO usar para tickets de queja (existe queue separado)."*

## Buenas prácticas de tool design

1. **Una responsabilidad por tool.** `create_notion_ticket` y `update_notion_ticket` separados, no `manage_notion`.
2. **Schemas estrictos.** Enums donde sea posible; required claros.
3. **Descripciones explícitas con casos de uso y contraejemplos.**
4. **Tools idempotentes.** Si el modelo lo invoca dos veces, no debe crear duplicados (usa idempotency key).
5. **Tools observables.** Cada llamada loguea inputs y outputs.
6. **Tools con timeouts.** Si una API tarda >5s, el agente bloqueado es mal agente.

## Multi-tool calling: paralelismo

Modelos modernos pueden invocar **varias tools en paralelo** en una sola decisión:

```
Modelo decide:
  - tool_use #1: search_kb("calendario exámenes")
  - tool_use #2: get_user_profile(user_id="andrea123")
  - tool_use #3: check_calendar_availability(week="2026-05-15")

App ejecuta los 3 en paralelo → resultados regresan juntos → modelo combina.
```

Útil cuando los pasos no dependen entre sí. Las plataformas no-code lo manejan automáticamente; en API tienes que coordinar tú.

## Tool choice forzada vs libre

- **Auto** (default): el modelo decide texto o tool.
- **Required**: el modelo debe usar **algún** tool.
- **Forced**: el modelo debe usar **este específico** tool.

Útil para arrancar pipelines: si quieres garantizar que el primer paso del agente sea una clasificación, forzar `classify_request` en la primera llamada.

## Errores frecuentes

- **Demasiados tools (>30).** El modelo se confunde, elige mal. Cura.
- **Schemas laxos.** El modelo inventa argumentos. Refuerza con enums.
- **Descripciones tipo `description: "tool x"`**. Inutilizables. Escribe en lenguaje rico.
- **Tools con efectos sin idempotency.** Reintentos crean duplicados.
- **Olvidar el caso `tool_error`.** Si la función falla, debes regresar el error al modelo para que reaccione, no callar.

## Tool use en plataformas no-code

- **n8n AI Agent node** → expones nodos del workflow como tools.
- **Dify Agent app** → marketplace de tools + custom tools.
- **OpenAI Assistants API** → file_search, code_interpreter, function tools.
- **Anthropic API** → tool use nativo + computer use.
- **Lindy** → tools predefinidos por categoría (email, calendar, CRM, etc.).

Las plataformas abstraen los pasos 1-4; tú solo defines tools y prompt.
