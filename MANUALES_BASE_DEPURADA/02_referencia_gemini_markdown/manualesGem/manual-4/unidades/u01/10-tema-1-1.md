# 1.1 Estructura XML y delimitadores

Le pides al modelo "haz un comunicado para padres de familia" y tres veces obtienes tres tonos distintos: una con emojis, otra con jerga corporativa, otra como circular escolar. ¿El modelo es voluble o tu prompt no tiene contrato? Este subtema convierte tu prompt en un contrato técnico.

::concepto{titulo="Estructura XML aplicada a prompts"}
Una **estructura XML en prompts** consiste en envolver cada componente del mensaje con etiquetas semánticas explícitas (`<role>`, `<context>`, `<task>`, `<constraints>`, `<examples>`, `<output_format>`). El modelo —especialmente Claude, GPT-5 y Gemini 2— está entrenado para reconocer esta estructura y tratarla como **secciones independientes** durante el razonamiento.

Las ventajas son tres:

1. **Parsing fiable** — el modelo distingue datos de instrucciones, instrucciones del rol, etcétera.
2. **Edición quirúrgica** — modificas una sección sin tocar el resto.
3. **Versionado y diff** — un prompt XML se versiona en Git como cualquier código.

XML no es la única estructura útil (también existen Markdown headings y delimitadores de tres comillas), pero es la más robusta cuando hay datos largos, múltiples instrucciones o riesgo de inyección.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Imagina que escribes una receta para un cocinero de turno y la pones toda en una sola línea sin separar ingredientes de pasos: "Tomas dos huevos sal pimienta los bates a fuego medio sirves caliente ensalada al lado." El cocinero **adivinará bien** el 60 % de las veces. Si en cambio la entregas con secciones (Ingredientes / Pasos / Tiempo / Notas), aciertas el 95 %. Tu prompt es exactamente esa receta. XML son las secciones.
::/interioriza

## Anatomía de un prompt XML profesional

El esqueleto que usarás para 90 % de los casos del Asistente Institucional:

```xml
<role>
Eres el Asistente Institucional Albatros. Tu tono es profesional, cálido
y respeta el manual de estilo del colegio.
</role>

<context>
Estás respondiendo a un padre de familia que pregunta sobre el calendario
de exámenes finales de bachillerato. La fecha actual es 28 de abril.
</context>

<task>
Redacta un comunicado de máximo 200 palabras con: fechas clave, lugar,
qué llevar, contacto para dudas.
</task>

<constraints>
- No inventes fechas que no aparezcan en el contexto.
- Usa tratamiento de "usted".
- No uses emojis.
- Si falta algún dato, di explícitamente "[dato pendiente — confirmar]".
</constraints>

<examples>
<example_good>
Estimadas familias: les compartimos el calendario...
</example_good>

<example_bad>
Hola padres! Aquí va lo de los exámenes...
</example_bad>
</examples>

<output_format>
Texto plano, sin encabezados markdown.
</output_format>
```

## Reglas de oro de las etiquetas

1. **Etiquetas semánticas, no decorativas.** `<task>` siempre dice qué hacer; `<context>` siempre da datos. No inviertas roles.
2. **Anidamiento permitido.** `<examples><example_good>...</example_good></examples>` es válido y útil.
3. **Cierre obligatorio.** `<task>...` sin `</task>` confunde al parser interno del modelo.
4. **Sin atributos decorativos.** No uses `<task priority="high">`; el modelo no los reconoce de manera estable.
5. **CDATA o triple comilla** cuando metes datos que pueden tener `<` o `>`: `<data>```...```</data>`.

## Delimitadores alternativos cuando XML estorba

| Caso | Delimitador recomendado | Por qué |
|---|---|---|
| Prompt muy corto (<50 palabras) | Markdown headings (## Tarea, ## Restricciones) | XML pesa demasiado |
| Pegas código fuente | Triple backticks (\```python ... \```) | Preserva indentación |
| Texto del usuario que no debe interpretarse como instrucción | `<user_input>...</user_input>` o triple comilla | Anti-inyección |
| Tabla de datos | Markdown table dentro de `<data>` | Legible para ti y para el modelo |
