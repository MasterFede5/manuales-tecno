# 1.8 Evaluación de prompts (rúbricas, A/B)

Tu compañera dice "el prompt v3 está mejor". Tu compañero dice "no, el v2 era más natural". Tú dices "me late el v4". Sin **datos**, esa discusión no termina. Este subtema te da las herramientas para que termine en 10 minutos.

::concepto{titulo="Evaluación de prompts"}
**Evaluación de prompts** es el conjunto de prácticas para **medir objetivamente** la calidad de un prompt antes de promoverlo a producción. Combina cuatro técnicas:

- **Golden set** — colección curada de inputs con outputs esperados o rúbricas.
- **Métricas automáticas** — exactitud, BLEU, F1, validez de schema, latencia, costo.
- **Rúbricas humanas** — calificación de calidad por criterios definidos.
- **A/B testing** — comparación en tráfico real entre versiones.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Cuando una farmacéutica lanza un medicamento nuevo no se basa en "se siente bien"; corre ensayos clínicos con grupos de control. Tu prompt afecta a 5 000 usuarios al mes. La diferencia entre v2 y v4 puede ser 10 % en satisfacción. **No tienes derecho a decidirlo a ojo.**
::/interioriza

## La pirámide de evaluación

```
                  ┌──────────────────┐
                  │  Eval humana     │  cada release
                  │  periódica       │
                  └──────────────────┘
                ┌──────────────────────┐
                │  A/B en producción   │  canary 5–10 %
                │  (tráfico real)      │
                └──────────────────────┘
              ┌──────────────────────────┐
              │  Golden set              │  pre-merge
              │  (20–50 ejemplos)        │
              └──────────────────────────┘
            ┌──────────────────────────────┐
            │  Smoke test                  │  cada cambio
            │  (3–5 ejemplos)              │
            └──────────────────────────────┘
```

Subes de nivel solo si pasaste el anterior. La mayoría del trabajo vive en **golden set**.

## Cómo construyes un golden set

1. **Recolecta 20–50 inputs reales** del histórico, **diversos** (típicos, raros, bordes).
2. **Para cada input, define el output ideal** o las **5 reglas** que debe cumplir.
3. **Versiona el set** en Git al lado del prompt: `evals/clasificador-v3.jsonl`.
4. **Automatiza la corrida**: script que pasa cada input por el prompt y compara con el ideal.
5. **Métrica por dimensión**: exactitud · cumplimiento de schema · latencia · costo.

## Rúbrica humana mínima (5 criterios, 1–5)

| Criterio | 1 | 3 | 5 |
|---|---|---|---|
| **Precisión factual** | datos inventados | parcial | impecable |
| **Adherencia al tono** | fuera de marca | aceptable | sello institucional |
| **Cumplimiento del formato** | malformado | con errores menores | estricto |
| **Utilidad para el usuario** | no resuelve | resuelve a medias | resuelve y orienta |
| **Manejo de bordes** | confunde fácil | maneja ambigüedad | escala con criterio |

Tres evaluadores califican 10 outputs. Promedias y comparas v3 vs v4. Si v4 supera v3 por ≥0.5 en al menos 2 criterios, **promueves**.

## A/B en producción (canary)

Cuando golden set no basta —porque el comportamiento real difiere—:

1. Despliega v4 al 5 % del tráfico, manteniendo v3 en el 95 %.
2. Mide durante 3–7 días: tasa de escalamiento humano, tiempo de resolución, encuesta de satisfacción.
3. Si v4 mejora todas las métricas o no empeora ninguna, sube a 50 %, luego 100 %.
4. Si empeora una métrica clave, **rollback inmediato**.

Herramientas: LangFuse, Helicone, PromptLayer, OpenAI Evals.

## Versionado y registro de cambios

Cada prompt vive en un repo (Git, Notion, Airtable). Convención mínima:

```
prompts/
├── clasificador-solicitudes/
│   ├── v3.0.xml
│   ├── v3.0.eval.json    ← golden set + scores
│   ├── v4.0.xml
│   └── CHANGELOG.md      ← qué cambió y por qué
└── redactor-comunicado/
    └── ...
```

CHANGELOG en lenguaje humano: *"v4.0 agrega `<de_escalation>` para reducir 30 % de quejas escaladas (ver eval). Breaking change: schema agrega campo `requiere_humano`."*

## Tres reglas de oro

1. **Nunca promueves sin golden.** Si no tienes golden set, lo construyes antes de la primera versión, no después.
2. **El equipo evalúa, no el autor.** Quien escribió el prompt no es el mejor para juzgarlo.
3. **El prompt en producción no es definitivo.** Programas re-evaluación trimestral. Modelos cambian, base de usuarios cambia, prompts envejecen.
