# 1.3 CoT avanzado y ReAct

Le preguntas a tu Asistente "¿cuántos cupos quedan en bachillerato si se inscribieron 47 de los 60 lugares y 3 dieron baja?" y responde "13". Está mal: la respuesta correcta es 16. ¿El modelo no sabe sumar? Sabe, pero no se tomó el tiempo. Este subtema enseña a forzar el tiempo de razonamiento.

::concepto{titulo="Chain-of-Thought (CoT) y ReAct"}
**Chain-of-Thought (CoT)** es una técnica de prompting en la que se instruye al modelo a **escribir explícitamente sus pasos de razonamiento intermedios** antes de dar la respuesta final. Reduce alucinaciones y errores en tareas que requieren cálculo, lógica o composición de pasos.

**ReAct** (de *Reasoning + Acting*) extiende CoT alternando tres elementos en bucle:
- **Thought** — qué piensa el modelo a continuación.
- **Action** — qué herramienta externa invoca (buscar, calcular, consultar).
- **Observation** — qué resultado obtiene.

ReAct es la base teórica de los agentes que verás en la Unidad 6. CoT trabaja solo con el modelo; ReAct lo conecta con el mundo.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Cuando un médico te explica un diagnóstico **mostrando su razonamiento** ("estos síntomas, descarto A por X, considero B porque Y…"), confías más que cuando suelta el diagnóstico final sin más. Lo mismo pasa con un modelo: la cadena visible de pensamiento es **la auditoría en tiempo real** de su decisión. Si la cadena tiene un error, lo ves; si la respuesta sale sin cadena, no puedes corregir nada.
::/interioriza

## CoT en su forma más simple

El gatillo clásico es la frase **"Piensa paso a paso"** al final del prompt. En modelos modernos eso ya casi no hace falta, pero el patrón estructurado sí mejora resultados:

```xml
<task>
Calcula cuántos cupos disponibles hay en bachillerato dadas estas reglas...
</task>

<thinking_format>
Antes de la respuesta final, escribe tu razonamiento dentro de <scratchpad>.
1. Identifica los datos.
2. Aplica la operación correcta.
3. Verifica con un método alternativo.
4. Solo entonces escribe la respuesta final.
</thinking_format>

<output_format>
<scratchpad>
[razonamiento paso a paso]
</scratchpad>
<answer>
[respuesta final concisa]
</answer>
</output_format>
```

Modelos de razonamiento (Claude Opus, OpenAI o3, Gemini Thinking) ya generan internamente este `<thinking>` y lo exponen en el output. Pero **forzarlo en el prompt** sigue siendo útil para auditarlo y para modelos más pequeños.

## CoT avanzado: las cuatro variantes que importan

| Variante | Cuándo usarla | Cómo |
|---|---|---|
| **Zero-shot CoT** | Tarea simple, modelo grande | "Piensa paso a paso." al final |
| **Few-shot CoT** | Tareas con formato específico | Incluyes 2–3 ejemplos resueltos con cadena visible |
| **Self-ask** | Preguntas compuestas | El modelo se hace sub-preguntas y se responde |
| **Plan-and-solve** | Problemas largos | Primero plan completo, luego ejecución por pasos |

Few-shot CoT es el que usarás más en producción: las cadenas de ejemplo enseñan al modelo el **formato exacto de razonamiento** que quieres.

## ReAct: el patrón Thought–Action–Observation

```
Pregunta: ¿Cuántos cupos quedan en bachillerato según el reglamento de
admisiones 2026?

Thought: Necesito el número total de cupos y los inscritos. El reglamento
está en la base de conocimiento.
Action: search_kb("cupos bachillerato 2026")
Observation: "60 cupos asignados a primer ingreso 2026."
Thought: Ahora necesito saber cuántos están ocupados.
Action: query_db("SELECT COUNT(*) FROM inscripciones WHERE periodo='2026'")
Observation: 47
Thought: Falta verificar si hubo bajas registradas.
Action: query_db("SELECT COUNT(*) FROM bajas WHERE periodo='2026'")
Observation: 3
Thought: Cupos disponibles = 60 - (47 - 3) = 16.
Final answer: Quedan 16 cupos disponibles.
```

Cada **Action** es una llamada real a una herramienta (búsqueda en base de conocimiento, consulta SQL, API externa). El modelo decide qué herramienta usar y con qué argumentos. Cuando profundices en agentes (U6) este loop es el corazón.

## Errores comunes y cómo blindarte

- **Cadenas falsas.** El modelo a veces inventa pasos plausibles que no son los que realmente usó. Mitigación: pídele dos cadenas distintas y compara (eso es self-consistency, subtema 1.4).
- **Loops infinitos en ReAct.** Si el modelo sigue invocando acciones sin converger, **límite duro** de pasos (ej. máx 8 iteraciones) y un Thought obligatorio "¿Tengo suficiente para responder?" cada 2 iteraciones.
- **Razonamiento expuesto al usuario final.** En producción, pones `<scratchpad>` y luego ocultas esa sección, mostrando solo `<answer>`. El proceso queda en logs para auditoría.

## Tres reglas para tu Asistente Institucional

1. **CoT por defecto** en cualquier tarea con cálculo o múltiples reglas (calificaciones, becas, plazos).
2. **ReAct cuando hay datos externos.** Cualquier respuesta que cite el reglamento o la base institucional usa ReAct con tools de búsqueda.
3. **Trazabilidad obligatoria.** Cada `<scratchpad>` se guarda con id de conversación y prompt-versión. Auditable a 30 días.
