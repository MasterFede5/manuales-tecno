# 1.7 Constrained output (JSON, schemas)

Tu Asistente clasifica una solicitud y entrega: *"La categoría es inscripciones, con confianza alta."* Bonito para un humano. **Inservible** para un sistema downstream que esperaba `{"categoria": "inscripciones", "confianza": "alta"}`. Este subtema cierra la grieta.

::concepto{titulo="Constrained output (salida restringida)"}
**Constrained output** es la práctica de forzar al modelo a producir respuestas en un **formato estructurado predecible** (típicamente JSON conforme a un schema), de manera que sistemas posteriores (bases de datos, APIs, otros prompts) puedan consumirlo sin parser frágil.

Cuatro niveles de garantía, de menos a más estricto:
1. **Prompt + ejemplo de JSON** — pedir "responde en JSON con esta forma".
2. **JSON mode** — flag del proveedor que garantiza JSON válido.
3. **Function calling / Tool use con schema** — el modelo llena un schema definido por ti.
4. **Structured Outputs estrictos** — modelo sigue el schema con 100 % de validez (OpenAI Structured Outputs, Anthropic Tool Use estricto).
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Cuando llenas un formulario médico en línea, hay campos: nombre, fecha, edad, síntoma. Si pudieras escribir cualquier cosa en cualquier orden, el sistema atrás colapsaría. **Los formularios son schemas.** En IA pasa igual: cualquier dato que tu modelo entregue a otra pieza de software (base de datos, n8n, app) tiene que llenar un formulario, no una redacción libre.
::/interioriza

## JSON Schema como contrato

Un schema describe forma, tipos, valores permitidos, campos obligatorios:

```json
{
  "type": "object",
  "properties": {
    "categoria": {
      "type": "string",
      "enum": ["inscripciones", "calificaciones", "becas", "queja", "otro"]
    },
    "confianza": {
      "type": "string",
      "enum": ["alta", "media", "baja"]
    },
    "tono_alerta": {
      "type": "string",
      "enum": ["neutro", "molesto", "agradecido"]
    },
    "requiere_humano": {"type": "boolean"}
  },
  "required": ["categoria", "confianza", "requiere_humano"],
  "additionalProperties": false
}
```

Pasas este schema al modelo (vía API: `response_format` en OpenAI, `tools` en Anthropic, `responseSchema` en Gemini) y el modelo **garantiza** que su respuesta cumple. Sin schema, el 5–10 % de las respuestas saldrían malformadas y romperían tu pipeline.

## Patrón de prompt para JSON

Aun cuando uses Structured Outputs, el prompt debe **describir cada campo**:

```xml
<task>
Clasifica la solicitud y devuelve JSON conforme al schema.
</task>

<schema_documentation>
- categoria: tema principal de la solicitud.
- confianza: tu certeza ("alta" cuando hay palabras claves directas;
  "media" cuando hay ambigüedad; "baja" cuando no hay señales claras).
- tono_alerta: estado emocional perceptible.
- requiere_humano: true si la situación pasa los límites del Asistente
  (legal, salud, queja grave).
</schema_documentation>

<input>
{texto del padre/madre}
</input>
```

## Mecanismos por proveedor

| Proveedor | Mecanismo (2025) | Garantía |
|---|---|---|
| OpenAI | `response_format: {type: "json_schema", strict: true}` | 100 % validez |
| Anthropic | Tool use con `input_schema` JSON Schema | 99 %+ validez |
| Google Gemini | `responseSchema` en API | Alta validez |
| Local (Ollama) | Outlines, Guidance, Instructor | Validez por construcción (gramática) |

## Casos de uso en tu Asistente

- **Clasificación** → JSON con enum de categorías.
- **Extracción** de datos de un correo → JSON con `nombre`, `email`, `tema`, `fecha`.
- **Generación de tareas para n8n** → JSON con `accion`, `parametros`, `prioridad`.
- **Respuestas con metadata** → JSON con `respuesta_humana`, `confianza`, `fuentes_citadas[]`.

## Errores típicos y blindaje

- **Schema demasiado laxo.** Si todos los campos son `string`, no validas nada. Usa `enum`, `pattern`, `minLength`.
- **`additionalProperties: true`** sin pensar. El modelo agregará campos que tu sistema no espera. Pon `false` por defecto.
- **No validar en runtime.** Aunque uses Structured Outputs, valida con un parser tradicional (Pydantic, Zod, ajv) antes de pasar al siguiente paso.
- **Schemas anidados muy profundos.** Más de 4 niveles raramente son necesarios. Aplana cuando puedas.

## Patrón "respuesta + metadata"

Lo más usado en sistemas reales: el modelo entrega **dos cosas a la vez**, una para humano, otra para máquina.

```json
{
  "respuesta_humana": "Las inscripciones para el ciclo 2026 abren el lunes 12 de mayo a las 9:00 en línea. Le enviaremos el enlace al correo registrado.",
  "metadata": {
    "categoria": "inscripciones",
    "confianza": "alta",
    "fuentes_citadas": ["reglamento_admisiones_2026.pdf"],
    "requiere_humano": false
  }
}
```

El frontend muestra `respuesta_humana`. El sistema usa `metadata` para logueo, métricas y enrutamiento.
