# 1.5 Prompt chaining

Pides al modelo en un solo prompt: *"Investiga el tema, redacta un comunicado, revisa el tono, agrega el formato oficial y corrige ortografía."* Sale algo aceptable, pero mediocre en todas las dimensiones. ¿Qué pasaría si en vez de un solo prompt monstruo usaras **cinco prompts especializados** encadenados? Eso es prompt chaining, y es como pasar de un cocinero que hace todo a una cocina con estaciones.

::concepto{titulo="Prompt chaining"}
**Prompt chaining** es la técnica de descomponer una tarea compleja en una secuencia de prompts más pequeños y especializados, donde **el output de cada prompt se convierte en input del siguiente**. Cada eslabón hace una sola cosa bien.

Tres patrones típicos:
- **Lineal** — Prompt 1 → Prompt 2 → Prompt 3 (cada uno transforma el anterior).
- **Ramificado** — un prompt genera N ramas paralelas que luego se fusionan.
- **Condicional** — el output de un prompt determina cuál es el siguiente (router).

Frente a un mega-prompt único, el chaining mejora calidad, controlabilidad y debugging.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
En un restaurante, hay un *prep cook* que pica verduras, un *line cook* que arma platos y un *expediter* que verifica antes de salir. Si juntas las tres tareas en una sola persona, el plato sale más lento y con más errores. Cada estación con un solo enfoque produce calidad superior. Tu Asistente Institucional sigue el mismo principio: cinco prompts especializados son superiores a uno polivalente.
::/interioriza

## Anatomía de una cadena lineal

Pipeline para producir un comunicado oficial:

```
INPUT: solicitud del director ("avisar suspensión de clases mañana")

PROMPT 1 — extracción de hechos
  IN: solicitud cruda + fecha + razones
  OUT: JSON con {fecha, motivo, alcance, contacto}

PROMPT 2 — estructura
  IN: JSON
  OUT: outline de 5 secciones (saludo, motivo, fechas, acciones, contacto)

PROMPT 3 — redacción
  IN: outline
  OUT: comunicado completo en tono institucional

PROMPT 4 — revisión de tono
  IN: comunicado
  OUT: comunicado revisado con notas de cambio

PROMPT 5 — formato final
  IN: comunicado revisado
  OUT: HTML/PDF con membrete, fechas, firma
```

Cada prompt tiene su propia plantilla XML, su propio versionado y su propia evaluación. Si la calidad cae, sabes exactamente **en cuál estación falló**.

## Patrón ramificado: paralelización

Cuando varias subtareas son independientes, las ejecutas **en paralelo**:

```
INPUT: documento de 20 pp del consejo académico

      ┌── Prompt A: extrae acuerdos de seguridad
      │
INPUT ┼── Prompt B: extrae compromisos académicos        ┐
      │                                                  │
      └── Prompt C: extrae fechas clave de calendario    │
                                                         ▼
                                              Prompt D: fusiona en resumen ejecutivo
```

Aceleras hasta 3× porque las llamadas A, B, C no se esperan entre sí.

## Patrón condicional: router

Un primer prompt clasifica la entrada y decide a qué pipeline ir:

```
INPUT: pregunta de un padre de familia

Router prompt (clasificador):
  → Si es "inscripciones" → pipeline-inscripciones
  → Si es "calificaciones" → pipeline-calificaciones
  → Si es "queja" → pipeline-quejas (con escalación humana)
  → Si es "otra" → pipeline-default
```

El router te ahorra tokens (no corres pipelines costosos cuando no aplican) y te permite **especializar** cada flujo.

## Reglas para diseñar cadenas robustas

1. **Una responsabilidad por prompt.** Si un prompt hace dos cosas, parte en dos.
2. **Contratos explícitos entre eslabones.** Cada paso recibe un formato y entrega otro definido (idealmente JSON o XML estructurado, ver subtema 1.7).
3. **Validador entre pasos.** Antes de pasar al siguiente, verifica que el output cumple el contrato. Si no, reintenta o pide corrección.
4. **Logueo total.** Cada paso loguea input, prompt-versión, output y tiempo. Sin esto, debugging es imposible.
5. **Costo por eslabón.** Mide tokens y latencia por paso. Cuando algo se vuelva caro, ese eslabón es candidato a optimizar o sustituir por código tradicional.

## Anti-patrones

- **Cadenas demasiado largas.** Más de 7 eslabones secuenciales suelen ser señal de que no descompusiste bien. Combina o paraleliza.
- **Pasar contexto innecesario.** Si el paso 4 no necesita el JSON crudo del paso 1, no lo pases. Cada token cuesta.
- **Modelos iguales en todos los pasos.** Para clasificar (paso 1) basta un modelo barato; para redactar el comunicado (paso 3) usas el modelo grande. Mezcla.
- **Cadenas sin idempotencia.** Si reejecutas el paso 4 con el mismo input dos veces, deberías obtener el mismo output (con `temperature=0`). Si no, debug.
