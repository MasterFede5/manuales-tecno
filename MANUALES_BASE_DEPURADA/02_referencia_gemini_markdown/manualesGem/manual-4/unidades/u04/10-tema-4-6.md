# 4.6 RAG vs fine-tuning: cuándo conviene

Tu Asistente sigue equivocándose con el lenguaje específico de tu institución ("expediente disciplinario", "comisión académica") aunque tenga RAG. Tu desarrollador propone fine-tuning. **Antes de gastar $500 en eso, este subtema te ayuda a decidir si realmente lo necesitas.**

::concepto{titulo="Fine-tuning"}
**Fine-tuning** es el proceso de **continuar el entrenamiento** de un modelo pre-entrenado con tus datos propios, ajustando sus pesos para que internalice estilo, formato o conocimiento específico. Resulta en un modelo "cousin" del original: igual de potente, pero sesgado hacia tu dominio.

Tres tipos:
- **Full fine-tuning** — todos los parámetros se actualizan. Caro, raro fuera de research.
- **LoRA / QLoRA** — solo se actualizan capas adicionales pequeñas. Estándar moderno.
- **Adapter / Prompt tuning** — capa muy ligera. Menos potente pero económico.

Distinguir bien: fine-tuning **no agrega "nuevos hechos"** al modelo de manera fiable. Agrega **estilo, formato, comportamiento**.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Cuando un nuevo cocinero entra a tu restaurante, no le rehaces el cerebro. Le **muestras los menús** (RAG) o le **das un curso de tu estilo** (fine-tuning). Mostrar menús es barato y al día siguiente puedes cambiarlos; el curso es caro pero, una vez tomado, funciona sin manual abierto. Ambas técnicas resuelven cosas distintas.
::/interioriza

## Comparativa nuclear

| Dimensión | RAG | Fine-tuning |
|---|---|---|
| **Qué resuelve** | Conocimiento factual específico | Estilo, formato, comportamiento |
| **Costo inicial** | Bajo (horas de setup) | Alto (USD 50-2000+ por entrenamiento) |
| **Costo recurrente** | Por chunks recuperados | Modelo separado, hosting o llamadas premium |
| **Actualización** | Re-indexar (minutos) | Re-entrenar (horas/días) |
| **Datos requeridos** | Documentos | Pares prompt-respuesta etiquetados (cientos a miles) |
| **Latencia por consulta** | +200-500 ms (recuperación) | Igual al modelo base |
| **Auditoría / explicabilidad** | Alta (citas) | Baja (pesos opacos) |
| **Riesgo de regresión** | Bajo | Medio-alto |
| **Control posterior** | Quitar documento → desaparece de respuestas | Pesos no se "olvidan" fácilmente |

## Cuándo USAR RAG (90 % de los casos institucionales)

- Conocimiento factual de tus documentos (reglamentos, calendarios, manuales).
- Cambia con frecuencia (semanal, mensual).
- Necesitas citas verificables.
- Tu equipo no tiene experiencia en ML training.
- Presupuesto limitado.

## Cuándo USAR fine-tuning

- **Estilo de escritura específico** que no se logra con prompts (tono institucional muy particular, lenguaje técnico, formato fijo).
- **Tareas repetitivas con formato fijo** (clasificación con etiquetas que no son del lenguaje natural, output JSON muy específico que el modelo confunde).
- **Dominio con jerga densa** (medicina, derecho, código aeroespacial) donde el modelo base produce muchos errores incluso con RAG.
- **Latencia crítica** (cuando RAG agrega 500 ms inaceptables en producción).

## Cuándo USAR ambos

Casos avanzados:
- Modelo fine-tuned para entender la **jerga** + RAG para los **hechos**.
- Ejemplo médico: modelo entrenado con jerga clínica que recibe historia clínica del paciente vía RAG.

## Por qué fine-tuning NO es la solución a "no sabe mi institución"

Es el error más común. Si el problema es "el modelo no conoce mi reglamento", fine-tuning con el reglamento como dataset:
- Cuesta $200-2000.
- Tarda 1-3 días.
- Cuando el reglamento cambia, vuelves a pagar.
- La calidad del recuerdo es **inferior** a leer el documento literal con RAG.
- No puede citar "página 18" porque internalizó el contenido sin metadata.

**RAG es objetivamente mejor para "que sepa mi institución".**

## Costos reales (2025)

| Proveedor | Tipo | Costo aproximado |
|---|---|---|
| OpenAI fine-tuning | gpt-4o-mini | $25/M training tokens + $7-30/M inference |
| OpenAI fine-tuning | gpt-4o | $25/M training tokens + $30-120/M inference |
| Anthropic fine-tuning | Claude (en beta para enterprise) | Caso por caso |
| Local (Ollama + LoRA) | Llama 3.1 8B | $0 GPU local, ~6 horas |
| RAG con commercial APIs | embedder + LLM | $50-200/mes a 50k consultas |

## Fine-tuning sin código (para curiosos)

- **OpenAI dashboard** (platform.openai.com → Fine-tuning) — subes JSONL, eliges modelo, lanza. UI simple.
- **Together AI** — interfaz no-code para LoRA con Llama, Mistral, Qwen.
- **Modal** — serverless training, plantillas listas.
- **Anthropic** — fine-tuning de Claude solo enterprise por ahora.

## Anti-patrones

- **Fine-tunear con datos que cambian.** Cada cambio = re-entrenar. Mejor RAG.
- **Fine-tunear con dataset pequeño (<100 ejemplos).** Resultado pobre o sobreajuste. Mínimo 500 buenos ejemplos.
- **Fine-tunear sin baseline.** Sin medir el modelo base con prompt + few-shot, no sabes si fine-tuning aportó.
- **Fine-tunear "porque suena bien".** Cuesta y lock-in. Justifica cada vez.

## Recomendación final para tu Asistente

1. **Fase 1-2** (90 % institucional): RAG.
2. **Fase 3** (cuando llegues): considera fine-tuning **solo** si el modelo base + RAG no logra:
   - Tono institucional muy específico.
   - Formato de salida que falla aun con few-shot.
3. **Antes de fine-tunear, prueba**: prompts más estrictos, more few-shot, modelos más grandes. Suelen resolver lo que parece "necesidad de fine-tune".
