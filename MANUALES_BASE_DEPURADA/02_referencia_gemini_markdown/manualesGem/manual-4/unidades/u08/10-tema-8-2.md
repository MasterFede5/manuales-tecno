# 8.2 Servidores MCP populares (filesystem, GitHub, Slack, Notion)

Cientos de servidores MCP existen. Esta lección cubre los **12 más útiles** que probablemente vas a usar desde el primer día.

::concepto{titulo="Catálogo de servidores MCP"}
Los servidores MCP se distribuyen como paquetes ejecutables (npm, pip, binarios) que arrancas con un comando. Anthropic mantiene un repo oficial con servidores curados, y la comunidad ha publicado cientos más en Hugging Face MCP Servers, Awesome MCP y otros catálogos.

Categorías:
- **Filesystem y archivos** (lectura/escritura local).
- **Versionado** (Git, GitHub, GitLab).
- **Comunicación** (Slack, Discord, email).
- **Productividad** (Notion, Airtable, Google Workspace).
- **Bases de datos** (Postgres, SQLite, MongoDB).
- **Web y búsqueda** (fetch, brave-search, browser).
- **Utilidades** (time, calculator, memory).
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Antes de adoptar IA en serio en tu institución, te pasarán por la mente 30 integraciones distintas que "tendría sentido conectar". Antes de MCP, eran 30 proyectos. Hoy, 25 de las 30 ya tienen servidor MCP listo. La pregunta cambia de "¿es factible conectar?" a "¿debería conectar?".
::/interioriza

## Top 12 servidores MCP imprescindibles

### 1. filesystem
**Comando:** `npx @modelcontextprotocol/server-filesystem /allowed/path`

**Tools:** `read_file`, `write_file`, `list_directory`, `search_files`, `move_file`.

**Caso del Asistente:** acceder a documentos institucionales en `/data/asistente-docs/`.

### 2. github
**Comando:** `npx @modelcontextprotocol/server-github` (requiere `GITHUB_PERSONAL_ACCESS_TOKEN` env).

**Tools:** `create_or_update_file`, `search_repositories`, `create_issue`, `create_pull_request`, `get_file_contents`.

**Caso del Asistente:** consultar prompts versionados (U1) en repo institucional, abrir issues de mejora.

### 3. slack
**Comando:** `npx @modelcontextprotocol/server-slack` (requiere `SLACK_BOT_TOKEN`).

**Tools:** `slack_post_message`, `slack_list_channels`, `slack_search_messages`, `slack_reply_to_thread`.

**Caso del Asistente:** notificar canal `#admisiones` cuando solicitud nueva, buscar conversaciones pasadas.

### 4. notion
**Comando:** servidores comunitarios (varios; el oficial está en desarrollo). Más usado: `mcp-notion-server`.

**Tools:** `create_page`, `update_page`, `search`, `query_database`.

**Caso del Asistente:** crear ticket en BD "Solicitudes" sin pasar por workflow externo.

### 5. postgres
**Comando:** `npx @modelcontextprotocol/server-postgres "postgresql://user:pass@host/db"`

**Tools:** `query` (SQL read-only típicamente).

**Caso del Asistente:** consulta directa a sistema de calificaciones interno (con permisos read-only).

### 6. sqlite
**Comando:** `npx @modelcontextprotocol/server-sqlite /path/to.db`

**Tools:** `query`, `read_query`, `write_query`.

**Caso del Asistente:** BD ligera de seguimiento de tickets de soporte.

### 7. brave-search
**Comando:** `npx @modelcontextprotocol/server-brave-search` (requiere `BRAVE_API_KEY`).

**Tools:** `brave_web_search`.

**Caso del Asistente:** búsqueda web actualizada (mejores prácticas pedagógicas, normativas externas).

### 8. fetch
**Comando:** `npx @modelcontextprotocol/server-fetch`

**Tools:** `fetch` (HTTP GET de URL).

**Caso del Asistente:** leer páginas públicas (calendario UNAM, notas educación).

### 9. memory
**Comando:** `npx @modelcontextprotocol/server-memory`

**Tools:** `create_entities`, `add_observations`, `search_nodes` (knowledge graph en memoria).

**Caso del Asistente:** dar memoria persistente al agente (perfiles de aspirantes, decisiones pasadas).

### 10. time
**Comando:** `npx @modelcontextprotocol/server-time`

**Tools:** `get_current_time`, `convert_time`.

**Caso del Asistente:** trivial pero útil para resolver fechas relativas correctamente.

### 11. google-drive (comunidad)
**Comando:** servidores varios; depende de cuál uses (OAuth requerido).

**Tools:** `gdrive_search`, `gdrive_read_file`.

**Caso del Asistente:** acceder al Drive del consejo académico directamente.

### 12. gmail (comunidad)
**Comando:** `mcp-gmail-server` u otro.

**Tools:** `gmail_send`, `gmail_search`, `gmail_read`.

**Caso del Asistente:** redactar correos sin pegar el contenido.

## Configurar un servidor en Claude Desktop

Edit `~/Library/Application Support/Claude/claude_desktop_config.json` (Mac) o equivalente:

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem",
               "/Users/me/AlbatrosDocs"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_xxxxx"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres",
               "postgresql://readonly:pwd@db.albatros.local/asistente"]
    }
  }
}
```

Reinicias Claude Desktop. Los servidores aparecen disponibles. Sin escribir código.

## Servidores custom (cuando lo built-in no alcanza)

Si tu sistema interno no tiene servidor MCP, lo construyes tú:

- **TypeScript SDK**: `@modelcontextprotocol/sdk` — el oficial.
- **Python SDK**: `mcp` (PyPI).
- **Plantillas**: el repo oficial tiene templates listos.

Construir un servidor MCP simple toma 1-2 horas para alguien con conocimiento básico de TypeScript/Python. Una vez hecho, todos los clientes MCP lo consumen.

## Estrategia de adopción para tu Asistente

**Mes 1**: filesystem + memory + time (los 3 más simples, valor inmediato).

**Mes 2**: GitHub (si versionas) + slack (si comunicas).

**Mes 3**: notion + postgres / sqlite (datos institucionales).

**Mes 6**: servidor MCP custom para tu sistema interno (calificaciones, expedientes).

## Anti-patrones

- **Conectar todo desde el día 1.** Comprométete a 1-2 servidores y crece.
- **Tokens con permisos overpowered.** Si tu Slack token tiene acceso a todos los canales, prompt injection es desastre. Mínimo necesario.
- **Servidor mcp sin actualización.** Bugs, security patches; mantén versiones.
- **No probar permisos.** Algunos servidores son "read+write" por default; valida que solo escriban donde permites.
