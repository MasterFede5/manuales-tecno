# 8.1 Model Context Protocol (qué resuelve)

Antes de noviembre 2024, conectar un LLM a tus archivos, GitHub, Slack y base de datos requería **4 integraciones distintas, cada una con su API y código**. MCP convirtió eso en un solo estándar. Esta lección te explica por qué importa.

::concepto{titulo="Model Context Protocol (MCP)"}
**MCP** es un estándar abierto creado por Anthropic en noviembre 2024 que define cómo los **clientes de IA** (Claude Desktop, Cursor, Windsurf, ChatGPT desde 2025) se comunican con **servidores que exponen herramientas, recursos y prompts**.

Arquitectura cliente-servidor:
- **Cliente MCP** — el LLM y su app (Claude Desktop, Cursor, etc.).
- **Servidor MCP** — un proceso separado (puede correr local o remoto) que expone:
  - **Tools**: funciones que el LLM puede invocar (`read_file`, `commit_to_git`, `query_db`).
  - **Resources**: contenido que el LLM puede leer (archivos, registros, URLs).
  - **Prompts**: plantillas reutilizables.

Comparable a USB: antes cada periférico tenía su conector; USB unificó. MCP es USB para integraciones LLM.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Antes de USB, cada impresora venía con cable y driver propio. Hoy, cualquier impresora conecta con cualquier computadora. MCP hace lo mismo con IA: una vez que un sistema (Slack, Notion, tu CRM) tiene servidor MCP, **cualquier cliente LLM puede usarlo** sin que tú escribas integración custom. Para tu institución, esto significa que tu Asistente puede integrarse a 30 sistemas en horas, no en meses.
::/interioriza

## El problema antes de MCP (Q3 2024)

Cada cliente LLM (ChatGPT plugins, Claude tools, Cursor extensions) tenía su propio mecanismo:

```
Pre-MCP:
  ChatGPT → plugin de OpenAI → integración X (custom)
  Claude  → tool use API     → integración X' (custom)
  Cursor  → MCP propio       → integración X'' (custom)
  ...
  
Cada integración construida 3 veces, una por cliente.
```

Cualquier startup que quisiera conectar su servicio a IA tenía que **escoger cliente** o construir 3 versiones. La fragmentación era enemiga del progreso.

## La solución MCP (desde Q4 2024)

```
Post-MCP:
  ChatGPT, Claude, Cursor, Windsurf, etc.  →  protocolo MCP estándar
                                                     ↓
                                          servidor MCP de [tu sistema]
                                          (escrito una sola vez)
```

Si tu sistema X expone servidor MCP, automáticamente lo usan **todos** los clientes que hablen MCP. La fragmentación desaparece.

## Anatomía de un servidor MCP

```typescript
// Conceptual; los servidores se escriben en cualquier lenguaje
server.tool("read_file", {
  description: "Lee contenido de un archivo en el filesystem permitido",
  inputSchema: { path: { type: "string" } },
  handler: async ({ path }) => {
    return fs.readFileSync(path, "utf8");
  }
});

server.resource("notion://page/{id}", {
  description: "Página de Notion",
  handler: async ({ id }) => {
    return await notion.pages.retrieve({ page_id: id });
  }
});
```

El cliente LLM lo descubre (`list_tools`, `list_resources`), decide qué invocar y maneja la respuesta.

## Tools, Resources, Prompts: las 3 cosas que un servidor MCP expone

| Capacidad | Qué es | Ejemplo |
|---|---|---|
| **Tools** | Funciones invocables (efecto en el mundo) | `commit_git`, `send_slack_message`, `query_db` |
| **Resources** | Contenido leíble (estado del mundo) | `file://path`, `notion://page/{id}`, `db://table/row` |
| **Prompts** | Plantillas reutilizables | "Resumir reunión usando contexto de [resource]" |

El cliente LLM decide cuándo invocar cada una basado en el objetivo del usuario.

## Transports

Los servidores MCP soportan distintos transportes:
- **stdio**: el cliente lanza el servidor como subproceso local. **Más común en 2025**, ideal para servidores locales.
- **HTTP/SSE**: el servidor corre como servicio HTTP, el cliente se conecta. Para servidores compartidos en red.
- **WebSocket**: alternativa para tiempo real.

Para tu institución, stdio (local) es lo más simple y privado.

## Anti-patrones

- **Construir tu propia integración cuando ya hay servidor MCP.** Reinventar es trabajo perdido.
- **Servidores MCP sin auth.** Si tu servidor MCP de DB no tiene auth, cualquiera con acceso al socket lo usa.
- **Mezclar tools genéricos con específicos.** Cura los tools del servidor: descripciones claras, nombres consistentes.
- **Olvidar la versión del protocolo.** MCP evoluciona; mantén actualizados clientes y servidores.
