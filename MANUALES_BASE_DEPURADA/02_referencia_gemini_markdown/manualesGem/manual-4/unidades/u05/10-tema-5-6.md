# 5.6 Integraciones con Notion, Airtable, Workspace

Donde tu equipo guarda info ahora mismo, ahí debe escribir tu workflow. Si los datos viven en Sheets pero tu workflow escribe en Notion, creaste **dos verdades**. Esta lección te enseña a integrarte con donde la verdad ya está.

::concepto{titulo="Integraciones de almacenamiento institucional"}
Las **3 plataformas dominantes** para guardar y operar datos institucionales sin BD relacional son:

- **Notion** — bases de datos integradas con docs, wikis y workflows colaborativos. Híbrido entre wiki y BD.
- **Airtable** — bases de datos relacionales con UI tipo spreadsheet, ideales para CRMs, inventarios, calendarios.
- **Google Workspace** — Sheets como BD ligera, Docs, Drive, Forms, Calendar. La opción más barata y universal.

Cada una tiene fortalezas distintas y APIs maduras que tus workflows consumen sin escribir backend.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Tu workflow brillante no vale nada si los datos terminan en un sistema que tu equipo no usa. La integración correcta es la que **escribe donde tu equipo ya lee**. Saber cuál es el "donde" es el 50 % del trabajo.
::/interioriza

## Notion — el wiki + BD

**Estructura:** páginas con contenido rico (texto, imágenes, embeds) + databases relacionales adentro.

**APIs nativas en plataformas no-code:**
- Notion in n8n: 12 acciones (create page, update page, get database items, append block, etc.).
- Notion in Make: módulo dedicado con 20+ operaciones.
- Notion in Zapier: trigger y actions.
- Notion in Power Automate: conector premium.

**Casos típicos del Asistente:**
- BD "Solicitudes de admisión" con propiedades (Status, Asignado, Fecha, Tipo, Notas).
- Wiki interna con artículos del FAQ alimentados por el equipo.
- Tablero kanban para flujo "Nuevo → En revisión → Aceptado / Rechazado".
- Calendario de eventos institucionales sincronizado con Google Calendar.

**Limitaciones:**
- API tiene rate limit (~3 req/s por integración).
- No es BD relacional verdadera; relations son menos potentes que Airtable.
- Performance baja con tablas >5 000 filas.

## Airtable — la BD relacional con UI amigable

**Estructura:** bases con tablas, vistas múltiples, relaciones entre tablas, fórmulas, automatizaciones internas.

**APIs nativas:**
- Airtable in n8n, Make, Zapier, Power Automate: muy maduras.
- Airtable web automations: si tu workflow vive dentro de Airtable, puede ejecutar antes de salir.

**Casos típicos del Asistente:**
- CRM de aspirantes con vistas por etapa.
- Inventario de aulas/equipos con disponibilidad.
- Calendario de exámenes con relaciones a profesores y aulas.
- Tracking de matrícula con lookups complejos.

**Fortalezas vs Notion:**
- Relaciones reales (lookups, rollups, formulas referenciadas).
- Mejor rendimiento con datasets grandes (10k filas sin problema).
- Vistas especializadas (Kanban, Gantt, Calendar) más maduras.

**Limitaciones:**
- Costo escala rápido (~$24/usuario/mes Pro).
- No tiene wiki integrado; complementario, no sustituto, de Notion.

## Google Workspace — la opción universal

**Estructura:** Sheets como BD ligera, Docs como wiki, Drive como almacén, Forms como input, Calendar como agenda.

**APIs nativas:**
- Google Sheets in n8n, Make, Zapier, Power Automate: extremadamente maduras.
- Google Drive, Docs, Calendar, Gmail: integraciones de primera categoría.

**Casos típicos del Asistente:**
- Sheet "Solicitudes_2026" con cada inscripción como fila.
- Forms público para captura inicial → trigger del workflow.
- Calendar para entrevistas con admisiones.
- Drive como repositorio de documentos.

**Fortalezas:**
- **Gratuito** en planes personales y muy barato corporativo.
- **APIs estables y rápidas.**
- **Acceso casi universal** (cualquier persona tiene Gmail).
- Forms como entrada gratuita de datos.

**Limitaciones:**
- Sheets degrada con >100k filas.
- Sin estructura nativa de "campos tipados" como Airtable.
- Wiki en Docs es menos expressivo que Notion.

## Patrón "tres ecosistemas"

Las instituciones serias **no eligen uno**: tienen un mapa.

| Tipo de dato | Dónde vive |
|---|---|
| Wiki con artículos del FAQ | Notion |
| BD de aspirantes y matrícula | Airtable |
| Forms de captura iniciales | Google Forms |
| Calendarios | Google Calendar (sincronizado) |
| Documentos largos (PRD, ADRs) | Notion + backup en Drive |
| Reportes | Sheets + Looker / Notion charts |
| Comunicación interna | Slack / Teams |

Tus workflows escriben en cada lugar según el dato. Sin el mapa, terminas duplicando.

## Patrón "single source of truth"

Para cada tipo de dato, **una única fuente**. Si la matrícula vive en Airtable, **nadie escribe matrícula directo a Notion**. Si Notion necesita mostrarla, lo hace vía sync (Notion sync con Airtable existe en mercado, hay servicios como Whalesync, Bardeen).

Cuando dos lugares contienen "la verdad" sobre lo mismo, **siempre divergen**.

## Setup mínimo de credenciales

En cada plataforma de workflow, configuras una credencial por servicio:

```
Credentials > Add new
  Notion → API token (Notion settings → integrations → create)
  Airtable → API key
  Google → OAuth (login con cuenta institucional)
```

**Práctica de seguridad:**
- Crea cuentas de servicio dedicadas para integraciones (no la cuenta personal).
- Limita permisos al mínimo necesario.
- Rota credenciales cada 6 meses.
- Documenta qué workflow usa qué credencial.

## Anti-patrones

- **Una sola plataforma para todo.** Notion para BD relacional grande sufre; Airtable para wiki es pobre.
- **Duplicar datos sin sync.** Crea "verdades paralelas" que divergen.
- **Sin convención de nombres.** "Solicitudes 2026", "Solicitudes_2026", "solicitudes-2026" → caos.
- **Ignorar rate limits.** Tu workflow falla en pico de inscripciones.
- **Permisos overpowered.** API token con admin de toda la org, suficiente para una sola tabla. Sobrepowered = riesgo.
