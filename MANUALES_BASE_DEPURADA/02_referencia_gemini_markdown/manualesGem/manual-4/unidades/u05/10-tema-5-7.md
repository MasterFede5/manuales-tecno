# 5.7 Patrones reutilizables

Después de construir 3 workflows, vas a notar **fragmentos repetidos**: un router de categorías, un retry con backoff, un mecanismo de idempotency. Reconocerlos como **patrones** y guardarlos como sub-workflows acelera la siguiente solicitud y eleva la calidad de todo el equipo.

::concepto{titulo="Patrones reutilizables en workflows"}
Un **patrón reutilizable** es un fragmento de workflow con una función específica y bien definida que se reusa en múltiples flujos: routing, retry, idempotency, fan-out/fan-in, approval, error handling. Las plataformas no-code permiten guardarlos como **sub-workflows** o **plantillas** que se invocan desde otros.

Construir biblioteca de patrones del equipo:
- Reduce tiempo de nuevos workflows (60-80 % menos).
- Eleva calidad (cada patrón fue probado en producción).
- Facilita onboarding (nuevo integrante hereda los patrones).
- Estandariza decisiones que de otra forma se reinventarían.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Cuando construyes tu segunda casa, no reinventas el sistema eléctrico: usas los mismos circuitos, breakers, normas. La construcción acelera y la calidad sube. Tu biblioteca de patrones de workflow es exactamente eso: la electricidad de tu institución, ya probada.
::/interioriza

## Los 5 patrones que rinden más

### Patrón 1 — Router por categoría

```
[Input] → [LLM clasificador] → [Switch/Router]
                                    ├── caso A
                                    ├── caso B
                                    └── caso default
```

Cuándo: cualquier flujo que recibe entradas heterogéneas. Sin router, los flujos crecen monolíticos.

Ejemplo: solicitud → categoría (inscripción / queja / info / otro) → 4 ramas distintas.

### Patrón 2 — Fan-out / Fan-in

```
[Input]
   ↓
   ├──→ [Tarea A: notificar]
   ├──→ [Tarea B: registrar]
   └──→ [Tarea C: agendar]
        ↓ ↓ ↓
        [Merge: agregar resultados]
            ↓
        [Output combinado]
```

Cuándo: necesitas hacer 3 cosas en paralelo y luego combinarlas. Acelera frente a secuencial.

Ejemplo: nueva inscripción → en paralelo (Notion + Gmail + Calendar) → merge → respuesta única.

### Patrón 3 — Retry con backoff exponencial

```
[Llamada API]
   ↓
   ¿exitosa?
   ├── sí → continuar
   └── no →
       ↓
       ¿reintentos < 3?
       ├── sí → wait 2^N segundos → reintentar
       └── no → fallback (notificar humano)
```

Cuándo: cualquier llamada externa. Sin retry, una API caída 30 segundos rompe tu día.

Ejemplo: Notion devuelve 500 → wait 2s, retry; sigue fallando → wait 4s; sigue → wait 8s; falla → Slack alert al equipo.

### Patrón 4 — Idempotency

```
[Webhook recibido con ID]
   ↓
[Hash o ID único = idempotency_key]
   ↓
[¿existe en cache/DB de procesados?]
   ├── sí → return 200 OK (ya procesado)
   └── no → guardar key → procesar → marcar procesado
```

Cuándo: cualquier webhook que pueda llegar duplicado (todos pueden). Sin idempotency, creas registros duplicados.

Ejemplo: Calendly envía webhook 3 veces por bug → solo procesas la primera, las otras 2 reciben "ya hecho".

### Patrón 5 — Approval loop

```
[Acción que requiere aprobación humana]
   ↓
[Mensaje a Slack/Teams con botones Aprobar / Rechazar]
   ↓
[Wait for response] (con timeout)
   ↓
   ├── Aprobado → ejecutar acción
   ├── Rechazado → notificar al solicitante
   └── Timeout → escalar a otro aprobador
```

Cuándo: acciones con consecuencia (cobrar, enviar comunicado público, eliminar datos). Humano en el loop antes de lo irreversible.

Ejemplo: comunicado redactado por IA → mensaje a coordinadora con botón "Aprobar y enviar" → si aprueba, Gmail; si rechaza, vuelve al editor humano.

## Cómo construir tu biblioteca

1. **Identifica.** Cuando termines un workflow, lista los fragmentos que parecen genéricos.
2. **Extrae.** Crea un sub-workflow con ese fragmento aislado, parámetros documentados.
3. **Documenta.** README breve con: qué hace, qué inputs, qué outputs, ejemplo de uso.
4. **Versionar.** Carpeta `workflows/patrones/` en tu repo Git con export JSON.
5. **Compartir.** Junta de 30 min con equipo presentando cada nuevo patrón.

## La regla del "tercera vez extrae"

- Primera vez que escribes algo: no lo extraigas, está OK inline.
- Segunda vez similar: copia y pega, anota mentalmente.
- Tercera vez similar: **extrae como patrón**. Es señal de que el equipo lo necesitará 10 veces más.

Extraer prematuro genera abstracciones malas (over-engineering). Extraer tardío deja deuda técnica. La tercera vez es el sweet spot.

## Anti-patrones de patrones

- **Patrones que nadie usa.** Construyes 12 patrones y solo 3 se reusan. Borra los 9 muertos.
- **Patrones sin documentación.** Inutilizables después de 2 meses.
- **Patrones acoplados.** Si un patrón solo funciona con cierta configuración del workflow padre, no es patrón: es spaguetti.
- **Versionar sin migration path.** Cuando actualizas un patrón, todos los workflows que lo usan podrían romper. Comunica.
