# 5.5 Webhooks, APIs y triggers

Tu workflow es perfecto. ¿Quién lo arranca? Si esperas a que alguien le dé "play", no es automatización: es asistencia. Esta lección cubre los 3 mecanismos que disparan workflows sin intervención humana.

::concepto{titulo="Triggers, webhooks y APIs"}
**Trigger** es el evento que hace que un workflow comience a ejecutarse. Tres familias principales:

1. **Schedule** — el workflow corre en un horario fijo (cada 5 min, todos los lunes a las 8:00, primer día de mes).
2. **Polling** — el workflow consulta periódicamente una API externa para detectar cambios ("¿hay correo nuevo?", "¿hay fila nueva en Sheets?").
3. **Webhook** — un sistema externo **notifica al workflow** cuando algo pasa. El workflow está "esperando" en una URL HTTP.

Una **API** es la interfaz programable de un servicio externo: una manera estándar de que tu workflow consulte o modifique datos en otro sistema (Notion API, OpenAI API, Calendar API). Casi todo workflow toca al menos una API.
::/concepto

::interioriza{titulo="¿Por qué te importa?"}
Sin saber elegir entre los 3 mecanismos, eliges siempre Schedule "por inercia": tu workflow corre cada 5 minutos consumiendo recursos sin que pase nada el 99 % del tiempo. Webhooks bien usados consumen recursos **solo cuando hay trabajo real**. La diferencia: $20/mes vs $200/mes en una operación pequeña.
::/interioriza

## Cuándo usar cada uno

| Caso del Asistente | Mejor trigger |
|---|---|
| Reporte semanal de uso | Schedule (lunes 8:00) |
| Detectar correo nuevo | Webhook si la app lo soporta; polling si no |
| Solicitud de inscripción desde formulario web | Webhook |
| Sincronización de Calendly con Google Calendar | Webhook (Calendly lo emite) |
| Backup de Notion | Schedule (semanal) |

| Cambio en una fila de Airtable | Webhook (Airtable lo soporta) |

## Webhook 101 — qué hay que saber

Un webhook es **una URL pública** que tu workflow expone. Cuando un sistema externo le hace `POST` con un payload JSON, el workflow se ejecuta usando ese payload como input.

```
URL del webhook: https://miinstitucion.app.n8n.cloud/webhook/admisiones

Sistema externo envía:
POST https://miinstitucion.app.n8n.cloud/webhook/admisiones
Content-Type: application/json
{
  "nombre": "Andrea Hernández",
  "email": "andrea.h@example.com",
  "mensaje": "Quiero información de inscripción para 1.º bachillerato 2026"
}

Workflow recibe → ejecuta → opcionalmente responde
```

**Seguridad mínima** del webhook:
- **URL larga e impredecible** (n8n las genera con UUID; nunca uses URLs adivinables).
- **Autenticación**: header `Authorization: Bearer <token>`.
- **Validación de payload**: verifica que tenga los campos esperados antes de procesar.
- **Rate limiting**: n8n y Make tienen límites; configura los tuyos.
- **HTTPS obligatorio.**

## API 101 — qué hay que saber

Cuando tu workflow llama a una API externa:

```
[HTTP Request node]
  Method: POST
  URL: https://api.notion.com/v1/pages
  Headers:
    Authorization: Bearer secret_xxx
    Notion-Version: 2025-09-03
  Body (JSON):
    {
      "parent": {"database_id": "abc"},
      "properties": {
        "Nombre": {"title": [{"text": {"content": "Andrea Hernández"}}]}
      }
    }
```

Las plataformas no-code te abstraen casi todo, pero te conviene entender:
- **Métodos HTTP**: GET (leer), POST (crear), PUT/PATCH (actualizar), DELETE (eliminar).
- **Headers**: identidad y formato.
- **Status codes**: 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 429 Rate Limit, 500 Server Error.
- **Pagination**: APIs grandes devuelven páginas; tu workflow debe iterar.
- **Rate limits**: cada API tiene un máximo (por ejemplo, Notion: 3 req/s).

## Patrones útiles

**1) Webhook + ack inmediato + procesamiento async.**
Si el procesamiento puede tardar, responde "OK recibido" rápido y procesa en background. Evita timeouts del cliente.

**2) Schedule + checkpoint.**
Si tu workflow programado consulta una API, guarda el último timestamp procesado. La siguiente corrida solo trae lo nuevo. Idempotencia.

**3) Polling con backoff.**
Si la API tarda en devolver cambios, multiplica el intervalo (1 min, 2, 4, 8…) hasta detectar uno. Vuelve a 1 min. Ahorra calls.

**4) Webhook con retry queue.**
Si tu workflow no puede procesar (downstream caído), encólalo y reintenta en 5/15/60 minutos. Las plataformas no-code suelen tener "Wait" + "If error".

## Anti-patrones

- **Schedule cada 1 minuto siempre.** Cara y derroche. Si nada cambia 99 % del tiempo, polling con backoff.
- **Webhook sin autenticación.** Cualquier bot en internet puede dispararlo y vaciar tu presupuesto.
- **No manejar 429 rate limit.** Tu workflow muere a la primera tormenta. Configura retry con exponential backoff.
- **Sin idempotency.** El mismo webhook llega dos veces y creas dos tickets. Usa el campo `id` del payload como deduplicación.
- **Hardcodear credenciales.** Usa el sistema de credentials de la plataforma.

## Caso aplicado: tu workflow institucional

```
[WEBHOOK] /admisiones (URL secreta, Bearer auth)
   ↓
[VALIDAR payload tiene nombre, email, mensaje]
   ↓
[VERIFICAR idempotency con email + timestamp]
   ↓
[CALL Anthropic API → clasificar mensaje]
   ↓
[CALL Notion API → crear ticket]  // 429 → retry
   ↓
[CALL Gmail API → enviar respuesta]  // 429 → retry
   ↓
[CALL Slack API → notificar canal]
   ↓
[Response 200 OK]
```

Cada llamada con retry, manejo de errores, logueo. El workflow funciona estable a 100 solicitudes/día.
